# 排布算法 v0.2 全链路说明

## 1. 文档目标

这份文档只描述当前仓库里已经存在的 footprint 排布算法。

目标不是提出新方案，而是把当前实际运行中的全套排布逻辑按函数级别拆开，回答四个问题：

1. 排布入口在哪里
2. 组几何和标签几何在哪里生成
3. 自动排布过程中哪些函数会修改组中心、标签朝向、标签偏移
4. 自动排布完成后，哪些函数只是消费结果，哪些函数又会再次改写结果

## 3. 当前算法的真实总链路

当前默认排布不是单阶段，而是 6 段串联：

1. 页面入口组装待排布组
2. 先给每组构造一个“初始碰撞几何”
3. 外层求解器给每组找最终组中心
4. 外层求解器结束后，按最终照片位置生成当前标签状态快照
5. 当前状态快照写回 `groupLayouts`
6. 渲染层完全依赖 `photos + groupLayouts` 重新生成最终显示几何

对应调用链如下：

1. `autoPlacePhotos(...)`
2. `solvePendingGroupPlacements(...)`
3. `placeGroupsLayerByLayer(...)` 或 `buildLegacyFallbackState(...)`
4. `buildGeometryMapForPlacements(...)`
5. `solveFrozenGroupLayouts(...)` + solver 最终几何偏移回写
6. `buildGroupGeometryFromLayout(...)`

这条链路意味着：

- 默认初始布局最终仍然通过 `photos + groupLayouts` 渲染
- 求解器先决定组中心，再参与最终标签偏移落地
- 当前状态快照阶段负责按最终组位置确定标签朝向，并继承 solver 已经算出的标签偏移

## 4. 数据结构

文件：`src/components/footprintLayoutTypes.ts`

### 4.1 `LogicalRect`

逻辑矩形：

- `left`
- `right`
- `top`
- `bottom`

用途：

- 照片包围盒
- 标签包围盒
- 线锚点包围盒
- 组合整体包围盒

### 4.2 `FootprintPlacement`

组中心坐标：

- `centerX`
- `centerY`

用途：

- 求解器只负责决定这个值

### 4.3 `PendingPlaceGroup`

待排布组：

- `placeKey`
- `placePhotos`
- `collisionGeometry`
- `collisionRect`
- `collisionScale`
- `reservedLabelOffset`
- `logicalX`
- `logicalY`
- `mapRect`
- `offsets`

关键点：

- `logicalX / logicalY` 是 POI 原始逻辑点
- `collisionGeometry` 是组的初始碰撞几何
- `collisionScale` 是当前排布预算使用的比例尺
- `offsets` 是组内照片相对组中心的偏移

### 4.4 `LockedPlaceGroup`

已存在且不参与本轮自动排布的组：

- `placeKey`
- `logicalX`
- `logicalY`
- `geometry`

### 4.5 `GroupLayoutSnapshot`

文件：`src/components/localMapGroupGeometry.ts`

当前标签状态快照：

- `placeKey`
- `labelSide`
- `labelOffset`

这是运行时渲染标签最关键的当前状态。

### 4.6 当前比例尺语义

当前系统里与排布相关的比例尺需要区分三类：

1. 全局缩放下限常量：`CLAMP_SCALE.min = 0.2`
2. fit view 动态算出的当前最小比例尺：`nextScale`
3. 页面当前实时显示比例尺：`outerScale`

当前真实语义：

- `0.2` 仍存在，但它只是交互缩放的理论下限，不是 footprint 排布预算基准
- 自动排布、保存冲突检测等默认链路，预算基准使用 fit view 动态算出的最小比例尺
- 手动拖动、手动整组调整等实时链路，预算基准使用当前页面的 `outerScale`
- 不允许再把固定 `CLAMP_SCALE.max` 当作“最大视图”预算值

## 5. 页面入口层：`page.tsx`

文件：`src/app/(shell)/user/footprints/page.tsx`

### 5.1 `getPhotoLogicalSize(photo)`

职责：

- 把图片真实宽高转成统一逻辑尺寸
- 后续所有组几何都依赖这个尺寸

### 5.2 `buildGridOffsets(...)`

职责：

- 按网格生成组内照片相对偏移

### 5.3 `buildStaggeredOffsets(...)`

职责：

- 按交错网格生成组内偏移

### 5.4 `buildRandomOffsets(...)`

职责：

- 随机生成组内偏移

### 5.5 `buildOffsetsForLayout(count, layout, cardSize)`

职责：

- 按预设布局模式选择上面三种 offset 生成器之一

### 5.6 `applySizedOffsets(placePhotos, offsets, gapX, gapY)`

职责：

- 根据每张图真实逻辑尺寸修正 offset
- 避免单纯按 cardSize 导致组内照片互相压住

### 5.7 `buildOffsetPhotoRect(placePhotos, offsets)`

职责：

- 把组内所有照片 offset 合并成一个照片外包矩形

### 5.8 `estimateReservedLabelOffset(placeKey, groupPhotos, scale, mapRect, existingLayouts)`

职责：

- 先用当前位置生成一次基础标签几何
- 再单组调用 `resolveGroupLabelLayouts(...)`
- 估算这个组应预留多少 `labelOffset`

注意：

- 这是页面入口层的“标签预留偏移估计”
- 这里不是最终手动状态

### 5.9 `solveFrozenGroupLayouts(photos, scale, mapRect, existingLayouts)`

职责：

- 在组中心都已经确定后
- 用最终照片位置重新生成每组基础几何
- 重新按最终组位置确定标签朝向
- 最终生成 `Map<placeKey, GroupLayoutSnapshot>`

这是默认初始布局里非常关键的一层。

### 5.10 `rebuildGroupLayoutSnapshotForCurrentPosition(...)`

职责：

- 用户手动拖动照片或整组后
- 直接根据当前照片位置重建单组 `GroupLayoutSnapshot`

当前现象里“手动挪一下就恢复”的行为，和这个函数直接相关。

### 5.11 `autoPlacePhotos(unplaced, referencePhotos, layout, options)`

职责：

- 这是普通自动排布的页面入口总函数

内部实际顺序：

1. 按 `placeKey` 分组未放置照片
2. 从 `poiPoints` 建立 `logicalPointByPlaceKey`
3. 扫描所有现有组，能锁定的组转成 `LockedPlaceGroup`
4. 对待排布组生成组内 offset
5. 计算 `reservedLabelOffset`
6. 计算 `preferredLabelSide`
7. 用 `buildGroupGeometryFromPhotoRect(...)` 先生成 `offsetGeometry`
8. 组装 `PendingPlaceGroup[]`
9. 调用 `solvePendingGroupPlacements(...)`
10. 把求解器输出的 `centerX / centerY` 回写到每张照片
11. 用最终照片位置调用 `solveFrozenGroupLayouts(...)`
12. 用 solver 最终几何回写 `labelOffset`
13. 把当前 `groupLayouts` 合并回页面状态

关键结论：

- `autoPlacePhotos(...)` 既负责组装输入，也负责生成默认状态快照
- 它不是单纯的“调用 solver 然后结束”
- 当前默认链路中，标签朝向由最终组位置决定，标签偏移继承 solver 最终结果

## 6. 组几何与标签几何层：`localMapGroupGeometry.ts`

文件：`src/components/localMapGroupGeometry.ts`

这一层是“所有标签/照片/锚点几何”的真实定义层。

### 6.1 基础工具函数

#### `rectsOverlap(a, b, gap)`

职责：

- 判断两个逻辑矩形在 gap 下是否重叠

#### `measureGroupLabelLayout(title, photoRectWidth, scale)`

职责：

- 计算组标签文字的宽高

#### `expandPhotoRect(photoRect)`

职责：

- 为照片区域外扩 padding

#### `buildPhotoRect(groupPhotos, getPhotoLogicalSize)`

职责：

- 从每张照片的 `frameX / frameY` 构造组合照片外包矩形

### 6.2 标签朝向规则

#### `resolvePreferredLabelSide(centerX, centerY)`

职责：

- 不看地图，仅按极角决定标签朝向

#### `resolvePreferredLabelSideForMap(centerX, centerY, mapRect)`

职责：

- 看地图矩形决定标签朝向
- 当前是“地图语义版”的标签上下侧判定核心函数

#### `isRectInLowerRegion(rect, mapRect)`

当前语义：

- 下方包围区不是“地图下边向两侧无限斜扩”
- 下方包围区也不是“地图中心横线以下”
- 下方包围区有且仅有一种定义：
  - 从地图中心出发
  - 经过地图左下角形成一条向外无限延展的射线
  - 经过地图右下角形成一条向外无限延展的射线
  - 两条射线之间的下方区域就是唯一包围区

当前实现要求：

- 组一旦落在这个包围区内
- 标签必须在组上方
- 不允许再被后续候选评分或包络判断改回下方

### 6.3 运行时包络扩展

#### `applyRuntimeEnvelope(geometry, mapRect)`

职责：

- 根据唯一分区规则，对 `groupRect` 和 `overallRect` 做额外上下扩张

注意：

- 它改的是碰撞/包络矩形
- 不直接改 `labelSide`
- 不直接改 `labelAnchor`
- 当前正确语义应是：
  - 分区命中对象必须与标签朝向判定使用同一参考对象
  - 不能一处看 `photoRect`，另一处看 `groupRect` 或已扩张后的 `overallRect`
  - 否则组会因为标签自身伸出而反过来改变自己所属区域

### 6.4 生成单组完整几何

#### `buildGroupGeometryFromPhotoRect(photoRect, title, photoCount, scale, fixedLabelSide, fixedLabelOffset, mapRect)`

职责：

- 从照片矩形直接生成完整 `GroupGeometry`

输出内容包括：

- `photoRect`
- `labelRect`
- `lineRect`
- `groupRect`
- `overallRect`
- `photoCenterX / photoCenterY`
- `labelSide`
- `labelAnchorX / labelAnchorY`
- `lineAnchorX / lineAnchorY`

这是整个排布系统里最关键的底层构造函数之一。

#### `buildGroupGeometry(groupPhotos, getPhotoLogicalSize, scale, fixedLabelSide, fixedLabelOffset, mapRect)`

职责：

- 先 `buildPhotoRect(...)`
- 再转调 `buildGroupGeometryFromPhotoRect(...)`

#### `createGroupLayoutSnapshot(placeKey, geometry)`

职责：

- 从 `GroupGeometry` 反推 `labelOffset`
- 记录成 `GroupLayoutSnapshot`

#### `buildGroupGeometryFromLayout(placeKey, groupPhotos, getPhotoLogicalSize, scale, layouts, mapRect, preserveSavedLabelSide)`

职责：

- 运行时消费 `groupLayouts`
- 从照片位置 + 布局快照重新生成最终显示几何

当前语义：

- 如果 `preserveSavedLabelSide === true` 且存在 layout
  - 使用 layout 里的 `labelSide`
  - 使用 layout 里的 `labelOffset`
- 否则
  - 重新调用 `resolvePreferredLabelSideForMap(...)`
  - `labelOffset` 回到 0

这是“当前布局状态消费入口”。

### 6.5 当前边界标签检测与比例尺预算

#### `getBoundaryLabelXMetrics(...)`

职责：

- 对边界过渡区标签做横向重合检测
- 为整组外推提供“还需要分离多少”的量化结果

当前真实语义：

- 默认自动排布链使用 fit view 动态算出的最小比例尺作为预算基准
- 手动实时链使用当前页面的 `outerScale` 作为预算基准
- 固定 `CLAMP_SCALE.max` 不再作为“最大视图”预算值
- `0.2` 也不再被当作默认排布预算值

### 6.6 标签候选集与当前状态求解

#### `buildGroupGeometryCandidatesFromPhotoRect(...)`

职责：

- 给固定 `photoRect` 生成一组标签候选几何
- 当前主链路里应只保留已确定朝向这一侧
- 可以有多个额外 offset 值
- 不应再构造另一侧候选

#### `buildGroupGeometryCandidatesFromGeometry(...)`

职责：

- 从已有几何再派生标签候选

#### `resolveGroupLabelLayouts(entries, options)`

职责：

- 在组中心固定的前提下
- 只对标签朝向和标签偏移做二次求解
- 最终返回每个组的 `GroupLayoutSnapshot`

当前内部关键步骤：

1. 按组距离中心等规则排序
2. 对每个 entry 生成标签候选集
3. 调 `pickBestResolvedCandidate(...)`
4. 调 `optimizeResolvedAssignments(...)`
5. 调 `searchResolvedAssignments(...)`
6. 用 `createGroupLayoutSnapshot(...)` 输出当前状态

当前正确边界应是：

- 只允许在既定朝向内求最小合法 offset
- 不允许改变 `labelSide`
- 不允许把“包围区上方标签”重新变成“组下方标签”

#### `pickBestResolvedCandidate(...)`

职责：

- 从候选标签几何里选择当前最优项

#### `scoreResolvedCandidate(...)`

职责：

- 为每个候选计算最终 score

当前关键评分项：

- 组间标签/照片重叠惩罚
- map overlap 惩罚
- 同侧近邻惩罚
- labelOffset 过大惩罚
- 与 baseGeometry `labelSide` 不同的翻面惩罚

### 6.7 仍在文件中的旧型整体求解函数

下面这些函数仍存在于同一文件，但当前主链路不是直接靠它们完成页面默认排布：

- `resolveGroupGeometryAsWhole(...)`
- `resolveGroupGeometryDownward(...)`
- `resolveGroupGeometryTextAware(...)`

它们更像保留的整体几何求解工具，不是 `page.tsx -> solvePendingGroupPlacements(...)` 主链的直接入口。

- 地图冲突
- 锁定组冲突
- 标签冲突
- 扇区平衡
- 整体包络惩罚

### 8.6 总入口

#### `solvePendingGroupPlacements(groups, mapRect, safeGap, labelGapBoost, lockedGroups, reportStage, reportMetric)`

职责：

- 当前默认自动排布的总入口

内部顺序：

1. `buildRadialLayout(...)`
2. `buildPlacementLayers(...)`
3. `placeGroupsLayerByLayer(...)`
4. 如果失败，转 `buildLegacySolverInputs(...)`
5. `buildLegacyFallbackState(...)`
6. `buildGeometryMapForPlacements(...)`
7. 返回 `placements + geometries + trace`

关键结论：

- `solvePendingGroupPlacements(...)` 的主目标是选组中心
- 它不直接产出最终 `groupLayouts`
- 它返回的 `geometries` 还会被页面层后续状态快照流程再次处理

### 9.3 分层主排布

#### `placeGroupsLayerByLayer(...)`

职责：

- 当前默认主求解路径

内部顺序：

1. 按 layer、pressure、radius 组织放置顺序
2. 给每个组构造 `preferredAngle / preferredRadius`
3. `findPlacementInField(...)` 找候选
4. 对每个候选调 `evaluatePlacementAgainstState(...)`
5. 如果 field search 失败，再走 slot fallback
6. 全部成功则返回 `PlacementState`
7. 某组失败则返回 `null`

### 9.4 精修阶段

#### `refineAnglesAndRadii(...)`

职责：

- 对已放下的组做小范围角度、半径微调

当前主入口 `solvePendingGroupPlacements(...)` 并没有直接显式调用这一步作为最终返回阶段，但相关 trace 和 refine 机制仍在文件内保留。

## 18. 当前算法 v0.2 的一句话总结

当前 footprint 排布算法不是“单次求解直接渲染”，而是：

`页面入口预构型 -> 外层求解器决定组中心 -> 当前状态快照再次生成 -> 运行时按当前快照重建显示几何`

因此任何标签方向问题，都必须区分到底出在：

- 初始待排布组构型
- solver 候选几何
- 当前状态快照生成
- 还是运行时当前快照消费

这四层不能混为一谈。

## 19. 当前设计口径修正

以下内容用于修正前文里与实际设计目标不一致的语义。

### 19.1 默认排布后的控制权

默认排布完成后，后续全部交给手动操作。

这意味着：

- 默认排布只负责产出首版合法结果
- 后续不应再自动替用户调整标签
- 如果文档中出现“默认排布后仍可能继续自动优化标签”的表述，应视为当前实现残留，不是设计目标

补充口径：

- `groupLayouts` 是当前布局状态快照，不是不可变冻结规则
- 手动操作后，标签应按当前位置实时重算
- 不应把默认阶段的标签朝向当作永久保留值

### 19.2 标签朝向的唯一规则

标签位置只有两种形态：

- 标签在组上方
- 标签在组下方

并且只有一种确定方式：

- 按分区确定

当前设计目标不是“候选评分后再决定哪一边更优”，而是：

- 先按区域分区确定标签朝向
- 确定后不再二次变化

### 19.3 分区规则的设计边界

上下左右分区应是唯一确定的基础规则。

其中当前已明确且必须固定的规则如下：

- 下方区域有且仅有一种定义
- 下方区域不是地图中心线以下
- 下方区域不是地图底边向外的模糊扩散带
- 下方区域就是：
  - 从地图中心出发
  - 穿过地图左下角的无限射线
  - 穿过地图右下角的无限射线
  - 射线之间向下无限延展的区域
- 处于这个包围区内的组，标签必须在组上方

其余区域定义如果未明确，不得靠补丁式 if 分支临时外推。

### 19.4 不允许二次决策

“翻面”不是设计概念，应从设计语言和算法语言中清除。

对于标签问题：

- 不允许在基础朝向确定后再次二次决策
- 不允许通过评分器在最终阶段把标签从一侧改到另一侧
- 不允许默认排布和手动状态使用两套不同的标签朝向决策链

这里的“不允许二次决策”特指：

- 默认自动排布完成后，系统不允许继续自动替用户改标签结果

不包括：

- 用户手动拖动后，按当前位置重新执行唯一分区规则

### 19.5 标签必须跟组走

标签是组的一部分，不是独立对象。

这意味着：

- 组中心确定后，标签随组整体移动
- 手动拖动组时，标签必须跟组走
- 默认排布和手动拖动的标签语义必须完全一致

进一步明确：

- 手动拖动后，标签位置按组的实际位置重新判断
- 不是沿用默认排布时的旧朝向
- 也不是独立拖标签

## 20. 基于设计口径反推当前实现存在的真实冲突

以下冲突是“当前实现”与“已确认设计口径”之间的冲突，不再把它们描述成开放问题。

### 20.1 默认排布后的标签落地链已做第一轮收敛

当前状态：

- 默认链路已不再把 solver 最终几何整包直接覆盖成单独重建结果
- 当前朝向与偏移已拆开处理：
  - 朝向按最终组位置确定
  - 偏移继承 solver 最终结果

剩余风险：

- 如果后续还有其他链路绕过这条写回规则，仍可能出现结果不一致

### 20.2 标签朝向目前并非由单一分区规则唯一决定

冲突位置：

- `resolvePreferredLabelSideForMap(...)`
- `applyRuntimeEnvelope(...)`
- `autoPlacePhotos(...)`
- `buildGeometryForPlacement(...)`
- `resolveGroupLabelLayouts(...)`

冲突原因：

- 当前实现里，标签朝向会在多个阶段被分别判断
- 其中还有包络、候选评分、状态快照层等额外影响层

与设计冲突点：

- 设计要求朝向只能由唯一分区规则决定

### 20.3 下方包围区语义曾被实现成其他几种错误语义

冲突位置：

- `resolvePreferredLabelSideForMap(...)`
- `isRectInLowerRegion(...)`
- `applyRuntimeEnvelope(...)`
- 手动拖动后的重建链路

已出现过的错误语义：

- 以地图中心横向下方为分界
- 以地图底边向左右做 45 度扩散带
- 以页面底边参与围成有限梯形
- 用过渡带补偿下方区域
- 用标签或整体扩张后的矩形反向定义所属区域

与设计冲突点：

- 这些都不是当前确认的唯一包围区定义

### 20.4 分区命中对象不统一会制造新漏洞

冲突位置：

- `resolvePreferredLabelSideForMapRect(...)`
- `applyRuntimeEnvelope(...)`
- `computeMapClearanceOffset(...)`

冲突原因：

- 一处用 `photoRect`
- 一处用 `groupRect`
- 一处再用扩张后的 `overallRect`

结果：

- 包围区边界附近的组会出现朝向与包络不一致
- 标签可能因为自身伸出而改变所属区域
- 默认排布与手动拖动会表现出不同的边界语义

与设计冲突点：

- 分区属于“组的位置语义”，必须有单一命中对象
- 不能被标签自身反向影响
### 20.5 “下部区域标签必须在组上方”未被收敛为系统级硬规则

冲突位置：

- `resolvePreferredLabelSideForMap(...)`
- `resolveGroupLabelLayouts(...)`

冲突原因：

- 即使某一阶段把下部区域判为“标签在上方”
- 后续默认状态快照阶段仍可能再按候选和评分改写

与设计冲突点：

- 下部区域标签在上方应是硬规则，不应被候选评分覆盖

### 20.6 默认排布与手动拖动的比例尺语义已分流统一

冲突位置：

- 默认：`autoPlacePhotos(...) -> solveFrozenGroupLayouts(...)`
- 手动：`rebuildGroupLayoutSnapshotForCurrentPosition(...)`

当前状态：

- 默认路径使用 fit view 动态算出的最小比例尺作为预算基准
- 手动路径使用当前页面 `outerScale` 作为实时预算基准

这两条链虽然输入比例尺来源不同，但语义已经一致：

- 默认链路看“当前默认视图”
- 手动链路看“当前实际视图”

### 20.7 标签目前仍被当作可独立优化对象

冲突位置：

- `estimateReservedLabelOffset(...)`
- `resolveGroupLabelLayouts(...)`
- `scoreResolvedCandidate(...)`

冲突原因：

- 当前实现把标签偏移和标签朝向都当作可额外优化的对象

与设计冲突点：

- 设计要求标签只有两种形态
- 设计要求标签位置只有一种确定方式
- 因此标签不应在基础排布后继续被“优化器”重新决策

### 20.8 额外高度与地图避让的对象一致性仍需持续检查

冲突位置：

- `applyRuntimeEnvelope(...)`
- `computeMapClearanceOffset(...)`
- `fitsGroupRectAroundMap(...)`
- `fitsLabelRectAroundMap(...)`

冲突原因：

- “是否属于下方包围区”
- “额外高度向哪一侧扩张”
- “地图禁入判断用哪个矩形”

这三件事当前虽然都和地图避让相关，但并不总是基于同一个几何对象。

当前状态：

- 默认排布主链、保存冲突检测、手动链的比例尺来源已不再混用固定常量
- 但额外高度、地图禁入、边界带横向冲突三者是否完全使用同一组最终几何，仍需继续跟踪

## 21. 按设计修复的目标定义

如果按当前设计修复，目标应收敛为下面 5 条，而不是继续做局部补丁。

1. 标签朝向只保留一个入口函数。

当前语义应是：

- 输入：组所在分区
- 输出：上方或下方

并且这个“组所在分区”必须明确：

- 以组照片主体对应的唯一参考对象命中分区
- 不能在不同阶段切换成 `groupRect`、`overallRect`、标签采样点

2. 默认排布阶段只决定一次标签朝向。

当前语义应是：

- 基础排布确定组中心
- 立即按分区确定标签形态
- 后续不允许再次改侧
- 包围区内一律是“标签在组上方”

3. 默认状态快照阶段不得再重选标签朝向。

当前语义应是：

- 当前状态层只允许保存默认结果
- 不允许再次“翻面”
- 不允许再次构造两侧候选集

4. 手动拖动与默认排布必须共用同一套标签规则。

当前语义应是：

- 默认排布得到的标签形态
- 与手动拖动后重建的标签形态
- 必须来源于同一分区规则
- 但手动阶段必须按当前位置重新计算，不能继承默认旧值

5. 标签必须被视为组的组成部分，而不是额外附属层。

当前语义应是：

- 组走，标签就走
- 组定侧，标签就定侧
- 标签不再拥有独立二次优化权

目标只有一个：

- 把标签朝向与标签位置收敛成单入口、单语义、单次决策

### 22.1 需要保留的唯一标签朝向入口

- 输入：组所在区域
- 输出：标签在上方或在下方

并且只允许它做：

- 分区规则判断

不允许它再承担：

- 候选比较
- 历史布局兼容判断

### 22.2 需要重新定义并冻结的区域规则

2. 下方区域固定为“地图中心穿过左右下角的两条无限射线”之间的区域
3. 朝向判定使用这套分区模型
4. 包络扩展如果仍保留，也必须使用同一套分区模型
5. 分区命中对象必须固定，不允许不同阶段切换

在这套定义冻结之前，任何继续围绕当前 if 分支补丁式调整的做法，都只能继续制造冲突。

### 22.3 页面入口层应做的收敛

文件：

### 22.5 标签二次决策层应被移除或降级

### 22.8 应删除的设计概念

以下概念应从后续设计与修复讨论中清除：

- “翻面”
- “根据候选评分自动选上或下”
- “默认排布后再自动修正标签方向”

保留的设计概念只应是：

- 分区
- 组
- 标签在上方/标签在下方
- 标签跟组走
- 默认排布后交给手动

### 22.9 推荐的最小修复顺序

如果按最小风险执行，顺序应是：

1. 先重定义唯一分区规则
- 输出明确的上下左右区域边界

2. 再收敛唯一朝向入口
- 所有朝向都只走 `resolvePreferredLabelSideForMap(...)`

3. 默认状态生成继续保持“朝向按最终组位置决定、偏移继承 solver 结果”
- `solveFrozenGroupLayouts(...)` 负责最终位置朝向
- solver 最终几何负责偏移落地
- `resolveGroupGeometryAsWhole(...)` 只允许在当前朝向内做整组避让

4. 最后统一默认与手动链路
- `autoPlacePhotos(...)` 与 `rebuildGroupLayoutSnapshotForCurrentPosition(...)` 共用同一规则
- 边界 5 度带标签横向冲突在 solver、layered placement、repair stages 中统一执行

### 22.10 按设计修复后的目标链路

按当前设计，理想链路应被收敛为：

1. 页面入口生成组内照片 offset
2. 求解器决定组中心
3. 唯一分区规则决定标签在上方还是下方
4. 组几何一次生成
5. 默认结果写成当前状态快照
6. 渲染层纯消费

也就是：

`offset 构型 -> 求组中心 -> 按分区定侧 -> 写入当前状态 -> 渲染`

而不是当前这种：

`offset 构型 -> 求组中心 -> 默认状态层再次决策标签 -> 渲染`

手动阶段应是另一条与其同规则的链路：

`用户拖动组 -> 按当前位置重算分区 -> 写入当前状态 -> 渲染`

### 22.11 当前已确认的新漏洞与冲突

1. 主求解器、分层放置器、修复阶段如果不共用同一条边界带规则，就会出现前一层判定生效、后一层又把组推回冲突位置的问题。

2. 只在 `groupRect` 上做地图避让会漏掉额外高度，尤其是下方区域和边界过渡区；地图禁入必须统一看 `overallRect`。

3. 后置整组几何求解如果还把标签候选当成上下双侧空间，就会把已确定朝向重新打乱，表现为“标签又回到下方”或“手动挪一下又恢复”。

4. 边界 5 度带不只是标签自身问题，而是整组排布问题；标签横向重合不足 `-10%` 时，必须推动整组向外，而不是只推标签。

5. 比例尺语义如果再次回退到固定常量，就会重新出现“预算已算但不是当前真实视图”的漏洞；默认链路必须继续使用 fit view 动态比例尺，手动链路必须继续使用当前 `outerScale`。
