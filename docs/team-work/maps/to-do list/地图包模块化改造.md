# 地图包模块化改造

## 目标

- 将当前地图能力按“地图类型”拆分为独立地图包
- 当前已知地图类型包括：
  - 普通地图
  - 铁路地图
  - 未来新增的专题地图
- 要求不同地图包之间代码完全隔离
- 后台管理界面的不同地图管理页，统一来源于各自地图包内部，而不是继续堆叠在公共管理目录中

## 当前问题

- 当前地图相关能力更偏向按接口或页面零散存放，不利于按地图类型独立演进
- 普通地图与铁路地图的业务边界、数据结构、后台管理能力没有形成包级隔离
- 后续如果继续增加地图类型，后台管理界面会持续膨胀，公共目录会承担过多分支逻辑
- 地图能力复用、差异能力扩展、权限控制、配置管理都缺少统一包边界

## 推荐方向

### 1. 以地图类型为一级模块

- 每一种地图都是一个独立包
- 每个地图包自行维护：
  - 前台页面能力
  - 后台管理界面
  - 接口
  - 业务服务
  - 配置
  - 类型定义

### 2. 建立 maps 容器层

- `maps` 只作为地图能力容器和注册中心
- 不再承载具体地图实现细节
- 它负责：
  - 地图包注册
  - 后台入口聚合
  - 公共协议定义
  - 通用能力抽象

### 3. 后台管理界面由地图包内提供

- 每个地图包导出自己的后台管理配置与页面组件
- 后台管理主界面只做装配和路由挂载
- 避免把所有地图管理页面直接写死在统一后台目录

## 后台挂载原则

- 每个地图包暴露统一注册对象，例如：
  - 地图标识
  - 地图名称
  - 后台菜单信息
  - 后台页面组件
  - API 路由前缀
  - 权限标识
- 后台容器根据注册对象渲染不同地图管理界面
- 新增地图时，原则上只新增地图包与注册信息，不修改大量公共后台代码

## 隔离原则

- 普通地图与铁路地图的数据模型、配置、服务逻辑不要互相直接引用
- 一个地图包内部改动，不应影响其他地图包编译和运行
- 公共能力只能放在 `maps/core`，且必须足够稳定、足够通用
- 如果某能力只服务某一种地图，必须留在该地图包内部

## 需要后续推进的事项

- 明确地图包注册协议
- 设计后台管理页面的统一挂载入口
- 识别普通地图与铁路地图的公共能力和专属能力
- 评估现有地图相关代码迁移成本
- 明确未来新增地图类型时的接入规范

## 结论

- 地图模块后续应从“接口集合”转为“地图包集合”
- 后台管理界面应从“公共目录集中堆放”转为“各地图包内部提供，后台统一挂载”
- `maps` 模块本身只保留容器、协议、注册和少量稳定公共能力
