# 本地目录索引方案 - 基于测试页验证收敛

> 已完成：测试页验证结论已迁移至正式 footprint 本地映射链路，完成归档见 `docs/team-work/footprint/done list/本地映射图片排列方案接入完成.md`

## 一、结论

在 `/test-css` 已完成一轮可运行测试后，`footprint` 的下一条正式方案应从“挂载网盘 / 云端同步”切换为：

- 用户在浏览器中手动选择一个本地主文件夹
- 主文件夹第一层子文件夹名即地点名
- 前端直接读取图片并按子文件夹分组展示
- 不上传原始图片
- 服务端仅记录目录索引摘要、排序位置和页面设置
- 二次进入时再次选择主文件夹
- 若主文件夹名称相同，则复用该主文件夹名称对应的已保存排序位置
- 若目录内容有差异，则提示新增 / 缺失 / 变化文件

该方案已通过测试页原型验证，当前结论是：

- 可行
- 比已回退的“网盘挂载”更贴近浏览器能力边界
- 第一阶段不应接数据库，先以轻量服务端记录跑通业务链路
- 后续若正式接入 `footprint`，建议再迁移到正式表结构

## 二、测试页已验证能力

当前测试页已验证以下能力：

- 用户可选择本地主文件夹
- 浏览器可读取主文件夹下图片文件
- 可按主文件夹第一层子文件夹分组展示
- 可进行组内手动排序
- 排序位置可通过服务端文本记录保存
- 二次打开后，重新选择同名主文件夹可恢复上次位置
- 可根据相对路径比对新增 / 缺失 / 变化文件

当前测试实现采用：

- 页面：`/test-css`
- 接口：`/api/test-css/session`
- 文本工作区：`test/test-css-workspace/folder-preview-session.json`

## 三、正式方案边界

### 1. 保留的能力

- 浏览器选择目录
- 前端直接预览本地图片
- 主文件夹名称作为记录键
- 第一层子文件夹名映射地点名
- 排序位置持久化
- 差异比对

### 2. 明确不做

- 不上传原始图片
- 不接入 `storage_files`
- 不复用 AList / 网盘挂载逻辑
- 不暴露真实绝对路径
- 不尝试后台自动扫描本地磁盘

### 3. 当前风险接受

- 若两个不同真实路径的主文件夹同名，会命中同一份记录
- 该规则是当前测试结论的一部分，第一阶段接受
- 若后续业务不能接受，再升级为“主文件夹名称 + 人工标识”或“句柄指纹”模式

## 四、正式接入 Footprint 的建议拆分

### 阶段 A：先替换测试实现为模块内原型

- 在 `footprint` 用户页增加“选择本地主文件夹”入口
- 复用测试页的目录扫描、分组展示、排序保存、差异提示逻辑
- 仍只走轻量服务端文本记录
- 不改数据库

### 阶段 B：模块化接口

- 将 `/api/test-css/session` 改造成 `footprint` 专用接口
- 把文本记录从“测试页会话”命名改成“本地目录索引会话”
- 保持记录内容只包含：
  - 主文件夹名称
  - 文件相对路径
  - 所属地点目录名
  - 文件大小
  - `lastModified`
  - 排序位置
  - 页面设置

### 阶段 C：再决定是否进数据库

- 若业务确认长期保留该能力，再设计正式表
- 正式表不应复用 `storage_files`
- 应单独建“本地目录索引”域

## 五、推荐的数据真相边界

第一阶段正式接入时，真相边界建议如下：

### 前端真相

- 当前用户本次选择的目录内容
- 当前会话中的图片预览 URL
- 当前拖动 / 排序中的临时状态

### 服务端文本真相

- 主文件夹名称
- 文件摘要列表
- 排序位置
- 页面设置
- 最近保存时间

### 暂不进入数据库的内容

- 原始图片
- 文件二进制
- 浏览器 blob URL

## 六、与旧方案的切割原则

后续文档与实现必须遵守：

- 不再沿用“挂载网盘”命名
- 不再引用 AList provider 作为第一阶段基础
- 不再让 `storage_files` 承担“未上传本地目录图片”的职责
- 不再将旧 `cloud_*` 草案视为当前第一阶段真相
