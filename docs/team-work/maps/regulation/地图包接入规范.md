# 地图包接入规范

## 目标

- 统一 `maps` 模块后续扩展方式
- 保证新增地图类型时，目录、前后台入口、接口归属保持一致
- 避免地图相关代码重新回流到公共目录堆叠

## 适用范围

- 普通地图
- 铁路地图
- 后续新增的任意地图类型包

## 模块结构规范

新增地图类型时，必须先在 `src/modules/maps/packages/` 下建立独立地图包。

示例：

```text
src/modules/maps/packages/
├── standard-map/
├── rail-map/
└── scenic-map/
```

每个地图包推荐包含：

```text
<map-package>/
├── admin/
├── api/
├── frontend/
└── index.ts
```

说明：

- `admin/`：地图包内部后台页面与后台相关导出
- `api/`：地图包内部接口业务逻辑
- `frontend/`：地图包内部前台能力，例如右侧栏、前台控制器
- `index.ts`：地图包主出口，注册地图包元信息

## 命名规范

- 地图包目录名统一使用英文短横线命名
- 示例：
  - `standard-map`
  - `rail-map`
  - `scenic-map`
  - `metro-map`

- 后台 URL slug 使用简短业务标识，不直接照搬包名
- 示例：
  - 包目录：`standard-map`
  - 后台 slug：`standard`

## 注册规范

每个地图包必须在 `index.ts` 中导出自己的注册对象。

至少包含：

- `slug`
- `packageName`
- `name`
- `description`
- `admin`
- `frontend`

示意：

```ts
export const scenicMapPackage = {
  slug: 'scenic',
  packageName: 'scenic-map',
  name: '景区地图',
  description: '景区点位与路线管理',
  admin: {
    enabled: true,
    entryPath: '/management/maps/scenic',
    page: ScenicMapAdminPage,
  },
  frontend: {
    rightPanel: ScenicMapRightPanel,
  },
};
```

新增地图包后，必须同步注册到：

- `src/modules/maps/core/registry/map-packages.ts`

补充说明：

- 代码注册表负责声明“这个地图包具备哪些前后台能力”
- 地图包是否在前台展示、后台默认顺序如何排列，当前以数据库表 `map_packages` 为准
- 新增地图包后，如数据库里还没有对应记录，应用层会按注册表自动补齐默认数据

## 后台规范

- 后台统一入口保留为 `/management/maps`
- 各地图后台页面必须来源于地图包内部
- 公共后台仅负责：
  - 列表入口展示
  - 根据 slug 挂载地图包后台页
- 后台地图包首页当前允许管理：
  - 启用 / 停用
  - 名称
  - 排序
- 后台地图包首页当前采用轻量卡片，不展示描述文案

禁止做法：

- 直接在 `src/app/management/maps/` 下继续堆叠具体地图业务页面
- 在公共后台目录内编写大量地图专属业务逻辑

## 前台规范

- 当前阶段前台 `/maps` 页面布局保持公共壳
- 地图包负责提供“自定义右侧栏”
- 当前不要求每个地图包接管整页前台布局
- 前台地图类型列表由公开接口 `/api/public/maps/packages` 提供
- 前台只展示“数据库已启用 + 代码已注册前台能力”的地图包
- 被停用的地图包必须在前台完全隐藏

当前约束：

- 普通地图包负责：
  - 搜索
  - 结果列表
  - 收藏
  - 加入足迹

- 铁路地图包负责：
  - 站点搜索
  - 搜索结果列表

后续如果要下沉左侧地图专属逻辑，必须单独评估后再推进，不要直接混改。

补充说明：

- 如果某类产品只是在同一底图内切换专题状态，例如当前的 `china-nature-map`，优先实现为“一个地图包 + 多个包内专题入口项”
- 不要在没有真实底图差异的情况下，过早拆成多个独立地图包

## API 归属规范

- `src/app/api/...` 只保留原始路由入口
- 具体业务逻辑必须写在地图包内部 `api/`

示例：

- 路由入口：
  - `src/app/api/maps/search/route.ts`

- 包内实现：
  - `src/modules/maps/packages/standard-map/api/search.ts`

禁止做法：

- 在 `src/app/api/maps/...` 中继续堆叠完整业务逻辑
- 在多个地图包之间互相直接复制接口逻辑

## 公共区准入规则

只有满足“多个地图都会稳定复用”的能力，才允许进入公共区。

可进入公共区的典型内容：

- 地图包协议
- 注册表
- 挂载逻辑
- 稳定通用类型

不应进入公共区的内容：

- 某个地图专属表结构
- 某个地图专属页面
- 某个地图专属接口逻辑
- 某个地图专属渲染算法

## 数据规范

- 能共享则共享，但不强行为了统一而统一
- 地图专属数据优先保留在各自地图包语义下
- 只有在多个地图明确复用同一种数据模型时，才考虑上收公共区

当前新增公共状态表：

- `map_packages`
  - 用于承载地图包名称、描述、启停状态、排序
  - 不承载 React 组件、页面实现、专属业务逻辑

当前允许的 package-local 轻量配置：

- 对于仍处在 UI-first 阶段、尚未接数据库的地图包，可以先在包内维护轻量配置源
- 例如：
  - `china-nature-map` 的专题入口项标题
  - 图片 URL
  - 排序
  - 启停
- 但这类配置应明确标注为过渡态，不要误写成长期正式数据层

## 新增地图包接入步骤

1. 在 `src/modules/maps/packages/` 下创建新地图包目录
2. 补齐 `admin/`、`api/`、`frontend/`、`index.ts`
3. 在包内实现后台页与前台右侧栏
4. 在包内实现接口业务逻辑
5. 在 `map-packages.ts` 中完成注册
6. 确认 `map_packages` 中已有对应记录，或由应用自动补齐后校验默认名称与排序
7. 如需协作文档，先按协作规则补充 `to-do list` 或个人记录

## 禁止事项

- 不要把地图专属逻辑重新写回公共后台目录
- 不要把地图专属接口重新写回公共 route 文件
- 不要绕过地图包注册直接硬编码地图入口
- 不要把“前台是否展示地图包”重新写死在 `/maps` 页面里
- 不要在未确认通用性的情况下，把专属能力塞进 `maps/core`

## 当前定稿边界

- 后台页面来源于地图包内部
- 前台右侧栏来源于地图包内部
- 地图接口业务逻辑来源于地图包内部
- 公共目录主要承担壳层、路由入口和注册职责
