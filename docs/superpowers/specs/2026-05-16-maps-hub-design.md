# 地图中心（Maps Hub）设计文档

## 背景

当前仓库已经存在多处地图能力，但它们都属于“局部嵌入”：

- `src/components/PlanMap.tsx`：高德地图基础封装，已支持标点、信息窗、聚焦、路线、覆盖物
- `src/app/(shell)/plan/page.tsx`：计划页左侧地图
- `src/app/(shell)/lists/page.tsx`：榜单页内嵌地图 + 点选联动
- `src/app/(shell)/user/footprints/page.tsx`：足迹页的地图/画布复合视图
- `src/app/api/markers/route.ts`：已存在 POI 标记点读取接口
- `src/db/schema.ts`：已存在 `markers` 表

现阶段需要把地图能力从“页面内功能”提升为“侧边栏一级入口”，形成统一地图中心，承载多种地图类型：

1. 普通地图
2. 中国铁路地图
3. 未来更多专题地图

## 本次目标

新增一个地图中心入口，放入左侧主导航，进入后展示统一地图页面。页面内部通过二级切换承载不同地图类型。

一期要求：

1. 侧边栏新增地图入口
2. 新建 `/maps` 路由
3. 支持至少两种地图视图
   - 普通地图：搜索、标注、查看已有 POI
   - 中国铁路地图：先以“可用/可展示/可参考”为目标
4. 地图类型切换结构可扩展，便于未来继续添加新地图

## 现状结论

### 已有能力可以直接复用

- 高德底图能力已经可用，不需要重新选型
- POI 数据结构已经存在，不必从零设计
- 侧边栏是统一配置式导航，新增入口成本低
- shell 布局已支持宽主区页面，适合承载地图中心

### 当前不足

- `PlanMap` 命名与职责偏“计划页专用”，不适合作为长期地图中心核心组件名
- 地图搜索能力尚未封装
- 标注能力目前偏“展示已有点”，缺少用户在地图上新增标注的交互闭环
- 中国铁路地图尚未在仓库内形成明确接入策略

## 路由与导航设计

### 一级入口


- `/plan` 后面，或
- `/footprints` 前面

## 普通地图设计

### 定位

普通地图是地图中心的一期主功能，优先保证它完整可用。

### 一期能力

1. 地图展示
2. 搜索地点
3. 展示现有标记点
4. 点击地图新增临时标注
5. 查看标注详情

### 搜索能力建议

优先基于高德 JS API / Web 服务做 POI 搜索与地址检索，不建议一期就切换到底层自建地理搜索。

建议支持：

- 关键词搜索
- 城市/区域限定
- 搜索结果列表
- 点击结果后地图聚焦
- 可从搜索结果直接创建标记

建议新增后端代理接口：

- `GET /api/maps/search?q=...&city=...`

原因：

- 避免把更多第三方 key 与调用细节散落到前端
- 便于后续切换服务商
- 便于缓存与错误处理统一化

如果一期为了提速，也可以先在前端直接使用 AMap 插件完成搜索，再在二期下沉到服务层。

### 标注能力建议

当前仓库已经有 `markers` 表和管理端，说明“系统级标记点”能力已具备基础。

一期前台地图建议分成两层标注：

1. 系统标注
   - 读取 `/api/markers`
   - 面向全站 POI 展示
2. 会话标注
   - 用户当前页面临时添加
   - 暂不入库，先验证交互

这样可以降低首版复杂度。等交互跑通后，再补：

3. 用户标注持久化
   - 新表或复用现有 marker 体系扩展 owner/source 字段

### 普通地图组件拆分建议

- `BaseAmapView`
  - 负责底图加载、地图实例管理、通用 marker/polyline/overlay
- `StandardMapView`
  - 负责搜索、结果联动、点击新增标注
- `MarkerEditorPanel`
  - 负责展示/编辑当前选中标注

### 关于 `PlanMap` 的处理

不建议继续把地图中心直接建立在 `PlanMap` 这个命名上。

建议分两步：

1. 先从 `PlanMap.tsx` 中抽出通用 AMap 容器逻辑
2. 新建更通用的组件，例如：
   - `src/components/maps/BaseAmapMap.tsx`
   - 或 `src/modules/maps/components/BaseAmapView.tsx`

然后：

- `plan`
- `lists`
- `maps`

都基于同一底层组件复用。

## 中国铁路地图设计

### 目标定位

中国铁路地图不建议一期就完全自研。

建议按三阶段推进：

1. 第一阶段：嵌入/承接外部成熟地图能力
2. 第二阶段：统一外层 UI 与站点/线路联动
3. 第三阶段：逐步沉淀自有铁路数据层与渲染层

### 第一阶段推荐策略

优先顺序如下：

1. 参考已有成熟项目的数据组织、图层表达、交互结构
2. 若对方页面允许嵌入，则以 `iframe` 或 WebView 风格做快速接入
3. 若不适合嵌入，则在我方页面提供“打开铁路地图”的承接视图，并保留未来替换能力

一期重点不是“复制整个铁路地图项目”，而是先让地图中心支持第二种地图类型，并验证用户路径。

### 接入边界

中国铁路地图与普通地图的差异很大：

- 底图可能不是通用 POI 导航视图
- 核心对象不是普通 marker，而是铁路线路、站点、网络关系
- 交互更偏专题信息图，而不是通用地点搜索

因此不要把它硬塞进普通地图同一套状态模型里。建议保持：

- 统一页面容器
- 分离地图视图实现

即：

- 共享 `MapsHubPage`
- 不强行共享内部业务逻辑

### 一期接入形态建议

`ChinaRailMapView` 建议支持三种模式之一：

1. `iframe` 内嵌模式
2. 外链跳转模式
3. 本地占位模式 + 后续接入

推荐优先级：

1. `iframe` 可用则优先
2. 否则先做外链承接

原因：

- 可以尽快形成真实可用的第二地图类型
- 不阻塞地图中心主框架建设

### 二期自建方向

如果后续要做自有中国铁路地图，建议独立建模：

- `rail_lines`
- `rail_stations`
- `rail_segments`
- `rail_station_aliases`
- `rail_service_types`

前端渲染方式建议优先考虑：

- SVG/Canvas 自绘专题图
- 或基于地图引擎叠加 GeoJSON 线路层

而不是继续只靠普通 marker。

## 风险与注意事项

### 1. `PlanMap` 职责膨胀

如果继续在 `PlanMap` 上叠加所有地图功能，它会同时承担：

- 地图底层引擎
- 搜索
- 标注
- 专题图逻辑
- 多页面兼容

后续会很难维护。需要尽快抽象分层。

### 2. 中国铁路地图版权/嵌入限制

外部成熟项目不等于可以自由复制其数据、代码或直接嵌入。落地前需要确认：

- 是否允许嵌入
- 是否有开源仓库与许可证
- 是否允许复用数据

### 3. 搜索服务依赖

若普通地图搜索依赖第三方服务：

- 需要考虑 key 暴露方式
- 调用频率限制
- 异常与空结果处理

## 推荐的最小落地版本

本轮开发最合适的 MVP：

1. 新增侧边栏“地图”入口
2. 新增 `/maps` 页面
3. 页面内支持“普通地图 / 中国铁路”切换
4. 普通地图复用高德底图，展示现有 `/api/markers`
5. 普通地图支持搜索框与地图点击临时标注
6. 中国铁路地图先做外部承接或嵌入占位

这样能快速验证产品结构是否成立，同时不把铁路专题开发复杂度提前引爆。
