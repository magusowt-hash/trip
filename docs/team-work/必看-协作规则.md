# 团队协作规则（必看）

## 目录结构

```
docs/team-work/
├── 必看-协作规则.md    ← 你正在看的文件
├── server/            ← 本机（服务端 / 主开发者）
├── mac/               ← 成员二
└── pc/                ← 成员三
```

## 协作流程

1. 各成员在自己的环境上完成分配到的开发/调试任务。
2. **一个模块的完整开发调试结束前**，各成员通过各自文件夹内的文档进行任务交接与合作。
3. 每个成员**每次结束自己的任务后**，需由我（项目负责人）**确认完成**。
4. 确认完成后，该成员需将本次改动摘要**写入文档到自己的文件夹内**。

## 文档命名规则

```
时间(精确到分)-标题.md
```

示例：
```
2026-05-16-1430-完善地图POI收藏接口.md
2026-05-16-1515-修复高德逆地理city字段空数组.md
```

- 时间格式：`YYYY-MM-DD-HHmm`（24小时制，精确到分钟）
- 标题：简要描述本次改动的主要内容

## 文档内容要求

每份文档至少包含：

```markdown
# 时间 - 标题

## 改动范围
- 列出涉及的文件

## 改动说明
简述做了什么，为什么这样做

## 验证方式
如何验证改动正确（测试步骤、截图、curl 命令等）

## 注意事项
- 依赖项、环境变量变更等
- 需要其他成员配合的事项
```

## Git 提交

- 文档写入后，必须执行 `git push` 推送到远程仓库，确保其他成员可拉取到最新状态。
- 提交信息格式建议：`docs(team-work): 2026-05-16-1430-完善地图POI收藏接口`

## 注意事项

- 每个成员**只在自己的文件夹内写入**，不要修改其他成员的文档。
- 确认流程不可跳过：未确认的改动不应提交文档。
- 文档尽量简洁，面向团队内部沟通，不必过度修饰。
