# Footprint 网盘挂载系统多形态可行性草案

## 目标

- 在不破坏当前 Footprint 显示方式的前提下，引入网盘挂载能力
- 支持严格的用户隔离
- 不将挂载系统与当前 OuterFrame 形态绑定，允许未来扩展出多种 Footprint 形态

## 结论

- 可行，但不应把“挂载目录”直接当作当前 Footprint 页面数据源
- 推荐将系统拆成“资源层 / 业务索引层 / 视图层”三层
- 当前 OuterFrame Footprint 只作为其中一种视图适配器
- 用户隔离必须放在服务端实现，不能依赖前端传路径

## 背景约束

### 现有文档基础

- `docs/superpowers/specs/2026-05-13-alist-image-display-design.md`
  - 已定义 AList 接入、按 `user_{id}` 目录隔离、`cloud_folder/cloud_cover` 等字段
- `docs/superpowers/specs/2026-05-14-footprints-outer-frame-design.md`
  - 当前 Footprint 显示方式依赖照片、地点关联、外框坐标 `frame_x/frame_y`
- `docs/superpowers/specs/2026-05-14-footprints-outer-frame-guide.md`
  - 当前页面和 API 依赖 `storage_files`、view token、文件访问权限控制

### 当前问题

- 网盘文件天然只提供“文件本身”，不天然提供：
  - 该文件属于哪个地点
  - 该文件在 OuterFrame 中的布局坐标
  - 该文件属于哪一种 Footprint 展示形态
- 如果直接让页面读取挂载目录，当前 Footprint 的显示逻辑无法稳定复用

## 推荐架构

### 1. 资源层

职责：统一描述“用户拥有哪些云端文件”，不关心具体 Footprint 形态。

建议字段：

- `id`
- `user_id`
- `provider`
  - 例如 `alist` / `webdav` / `local-mount`
- `root_id`
  - 某个挂载根或空间标识
- `relative_path`
- `file_name`
- `mime_type`
- `size`
- `etag` 或 `version`
- `thumb_url`
- `origin_url`
- `captured_at`
- `last_seen_at`
- `status`
  - `active / missing / forbidden / deleted`

说明：

- 资源层不存当前页面专用字段，如 `frame_x/frame_y`
- 资源层只解决“文件发现、访问、缓存、同步、权限”

### 2. 业务索引层

职责：把云端文件映射为 Footprint 业务对象。

建议字段：

- `asset_id`
- `user_id`
- `place_title`
- `place_id`
  - 若后续地点标准化，可补充
- `group_id`
- `cover_priority`
- `tags`
- `visibility`
- `binding_source`
  - `auto / manual / imported`

针对不同形态增加扩展字段：

- OuterFrame 形态
  - `frame_x`
  - `frame_y`
  - `layout_version`
- 时间轴形态
  - `timeline_at`
  - `timeline_bucket`
- 地图聚合形态
  - `cluster_key`
  - `marker_style`

说明：

- 这一层才承接当前 Footprint 的地点、分组、封面、布局信息
- 不同形态共享同一批资源，但不共享完全相同的布局字段

### 3. 视图层

职责：将业务索引层数据投影为具体页面所需结构。

当前至少会有：

- `OuterFrame Footprint`
  - 需要 `place_title + frame_x/frame_y + 图片列表 + POI 连线关系`
- 未来可能新增：
  - 时间轴 Footprint
  - 相册墙 Footprint
  - 地图聚类 Footprint
  - 故事流 Footprint

说明：

- 视图层不直接读取挂载目录
- 视图层只消费后端已经整理好的业务数据

## 对“当前显示方式”的兼容策略

### 可兼容方式

保持当前页面读取的仍然是“受控照片记录”，只是这些记录可以来自两类来源：

- 本地上传资源
- 云端挂载同步资源

对于当前 OuterFrame 形态，页面关心的仍是：

- 照片标识
- 地点标题
- 可访问图片 URL
- `frame_x/frame_y`

因此兼容关键不在“是否挂载”，而在于后端是否能把挂载文件整理成等价记录。

### 不推荐方式

- 页面直接读取 AList 或挂载目录路径
- 用目录层级硬编码地点含义
- 用文件系统结构直接承载布局坐标

原因：

- 目录结构不稳定
- 多形态时会迅速失控
- 权限边界会变弱

## 用户隔离方案

### 原则

- 一切路径拼接都在服务端完成
- 前端只传资源 ID 或相对路径
- 任意请求都必须绑定当前登录用户

### 推荐实现

#### 方案 A：逻辑挂载优先

- 底层通过 AList 或其他网盘 API 访问文件
- 后端按 `user_id` 自动注入用户空间根目录
- 例如：
  - `/root/user_123/...`
  - `/root/user_456/...`

优点：

- 与当前文档一致
- 易于做路径校验
- 易于签名 URL 和访问审计

#### 方案 B：真实挂载也必须包一层服务端资源网关

即使未来把网盘真实挂到服务器目录，也不能让业务侧直接按绝对路径读文件。

仍应通过统一资源服务访问：

- 校验 `user_id`
- 校验路径是否落在该用户挂载根内
- 屏蔽 `../`、符号链接逃逸、跨目录引用
- 对外只暴露受控文件 ID 或短时签名链接

### 必须防的风险

- 用户构造路径访问他人目录
- 目录重命名导致旧绑定失效
- 短时签名 URL 泄露后长期可访问
- 挂载扫描结果与数据库索引不一致

## 与未来多形态 Footprint 的边界

### 稳定底座

以下能力应作为长期稳定底座：

- 用户文件发现
- 用户资源隔离
- 文件访问鉴权
- 缩略图与原图获取
- 资源同步与失效标记
- 地点绑定关系

### 不应下沉到底座的内容

- OuterFrame 的 `frame_x/frame_y`
- 某种页面的排序逻辑
- 某种页面专用标签样式
- 某种页面的交互状态

说明：

- 这些属于具体形态的业务或视图扩展，不能反向污染挂载系统

## 数据模型建议

### 方向 1：复用 `storage_files`，最小改造

做法：

- 将挂载文件同步为 `storage_files` 记录
- 增加来源字段，例如 `source_type = local | cloud`
- 保留当前页面依赖的 `place_title / frame_x / frame_y`

优点：

- 当前 Footprint 复用成本最低
- 现有 API 和页面改动较少

缺点：

- `storage_files` 会逐步承担过多职责
- 未来多形态扩展时容易继续耦合

适用：

- 近期目标是尽快接入挂载能力并跑通当前页面

### 方向 2：新增云端资源索引表，长期更稳

做法：

- 新增资源索引表承接挂载文件
- 通过映射表或投影层输出给当前 Footprint
- 当前页面继续消费整理后的统一照片模型

优点：

- 更适合多形态演进
- 资源层和视图层边界清晰

缺点：

- 首期改造成本更高
- API 整理工作更多

适用：

- 目标不仅是补充相册，而是把网盘资源作为长期基础设施

## 推荐决策

### 短期

- 采用“逻辑挂载 + 服务端索引”的方向
- 保持当前 OuterFrame 形态可继续使用现有照片模型
- 不直接暴露真实挂载目录给前端

### 中期

- 将“云端资源访问”从当前 Footprint 页面里抽离成独立资源能力
- 明确资源层与布局层的边界

### 长期

- 若 Footprint 形态继续增多，应逐步从“复用 storage_files”过渡到“独立资源索引层 + 多视图投影”

## 待确认问题

- 当前是否接受把云端文件同步进 `storage_files`
- `place_title` 由目录推断、文件名推断，还是人工绑定为主
- 目录重命名后，绑定关系是按路径维护还是按资源指纹维护
- 未来多形态 Footprint 是否共享同一套“照片分组”定义

## 当前建议

- 如果目标是尽快落地，不要先做真实文件系统挂载，先沿用现有 AList 思路扩展
- 如果目标是长期基础设施，不要让当前 OuterFrame 结构反向定义整个挂载系统
- 挂载系统的定义应是“用户资源接入能力”，不是“当前 Footprint 相册实现”
