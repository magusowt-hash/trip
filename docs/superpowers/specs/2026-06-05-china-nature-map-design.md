# 中国自然地图设计文档

## 背景

当前 `maps` 模块已经具备地图包注册、后台地图包启停、前台地图类型列表受后台驱动等基础能力，但自然主题地图还没有独立产品形态。

这次不直接做真实自然地理数据与点位系统，而是先把“中国自然地图”的前后台壳层、专题入口形式、后台自定义能力和专题切换结构做出来，为后续接入真实标注数据留出稳定骨架。

## 本次目标

新增一个前台地图包：

- `china-nature-map`

它的职责不是切换底图，而是在同一地图壳内承载多个自然主题入口与专题切换状态。

本次目标包括：

1. 在 `/maps` 中新增“中国自然地图”入口
2. 进入后展示自然主题前台列表
3. 前台列表采用单列大图流，每项只展示图片与标题
4. 图片与标题完全由后台自定义
5. 点击专题项后进入同一底图下的专题地图视图壳
6. 子主题当前只用于切换专题状态，不进行底图切换
7. 一期不接真实自然点位与专题标注数据

## 不在本次范围内

- 不做真实自然地理点位
- 不做专题标注绘制系统
- 不做子主题独立地图包底图
- 不做图片上传系统
- 不做描述、副标题、标签等额外字段
- 不做复杂后台批量编辑

## 产品定位

“中国自然地图”一期定位为：

- 旅行选题地图入口
- 通过视觉化专题卡片帮助用户按自然主题进入地图
- 后续承接海岛、喀斯特、雅丹等专题的真实标注系统

它不是知识图谱页，也不是独立多底图系统。

## 信息架构

### 地图包层级

本次采用：

- 一个总地图包：`china-nature-map`
- 多个总包内部的专题配置项

当前不把海岛、喀斯特、雅丹实现成真正独立地图包，而是先作为总包内部的“专题入口项”存在。

原因：

- 用户当前要的是专题切换，而不是底图切换
- 一期只做 UI 轮廓，不做真实数据
- 这样可以更快把前后台交互结构跑通

### 前台结构

建议前台拆成两个层次：

1. 中国自然地图专题列表页
2. 中国自然地图专题地图页

其中：

- 列表页展示后台维护的专题入口卡片
- 专题地图页保持现有地图壳结构，但只体现当前专题状态

## 前台设计

### 1. 专题列表页

专题列表页采用单列大图流。

每项卡片只展示：

- 背景图
- 标题

不展示：

- 描述
- 标签
- 数量
- 副标题

### 2. 卡片视觉

每张卡片：

- 图片占据主体
- 标题压在图片内部底部
- 标题容器采用半透明液体玻璃风格
- 不额外占据卡片外部空间

视觉目标：

- 更像自然旅行专题封面
- 不像后台管理卡片或信息列表

### 3. 专题地图页

点击卡片后进入专题地图页。

该页仍然使用统一地图壳，不切换到底图新页面。

当前页内需要体现：

- 当前专题标题
- 当前专题已选中状态
- 可回到专题列表
- 地图区域先保留空底图

一期专题切换行为：

- 只切当前专题状态
- 不切底图
- 不加载真实专题标注

## 后台设计

### 1. 管理对象

后台本次管理的是“中国自然地图专题入口项”，不是地理点位。

### 2. 后台字段

每个专题入口项至少包含：

- `topicSlug`
- `title`
- `coverImageUrl`
- `sortOrder`
- `isEnabled`

说明：

- `title`：前台卡片标题
- `coverImageUrl`：前台卡片大图 URL
- `sortOrder`：列表顺序
- `isEnabled`：前台是否显示
- `topicSlug`：专题唯一标识，用于进入专题地图页后标识当前主题

### 3. 后台交互

后台首页先支持：

- 查看专题入口项列表
- 编辑标题
- 编辑图片 URL
- 编辑排序
- 启用 / 停用

当前不要求：

- 上传图片
- 批量导入
- 描述字段编辑

## 数据设计

建议新增一张“中国自然地图专题入口配置表”，例如：

- `nature_map_topics`

字段建议：

- `id`
- `topic_slug`
- `title`
- `cover_image_url`
- `sort_order`
- `is_enabled`
- `created_at`
- `updated_at`

说明：

- 该表只承载专题入口配置
- 不承载地图组件能力
- 不承载真实地理点位

## 路由建议

建议保持在 `china-nature-map` 地图包下完成：

- 后台管理页：
  - `/management/maps/china-nature`
- 前台入口：
  - `/maps` 中的“中国自然地图”

总包内部前台子页面建议有两种状态：

- 专题列表状态
- 专题地图状态

可通过包内状态切换实现，不必额外为一期建立复杂子路由。

## 组件建议

建议在 `china-nature-map` 内部拆分：

- `NatureTopicListView`
- `NatureTopicHeroCard`
- `NatureTopicMapView`
- `NatureTopicSwitcher`

其中：

- `NatureTopicListView` 负责大图流列表
- `NatureTopicHeroCard` 负责图片+标题玻璃条样式
- `NatureTopicMapView` 负责专题地图页壳层
- `NatureTopicSwitcher` 负责当前专题切换状态

## 实施顺序

建议按下面顺序推进：

1. 建立 `china-nature-map` 地图包骨架
2. 接入 `maps` 注册表
3. 做前台专题列表页 UI
4. 做专题地图页壳层 UI
5. 做后台专题入口项管理页 UI
6. 再决定是否接数据库与真实配置接口

## 风险与边界

### 1. 结构风险

如果现在就把海岛、喀斯特、雅丹都拆成真正独立底图包，会和“只切专题、不切底图”的产品方向冲突。

因此一期要明确：

- 这些是专题入口项
- 不是独立底图页

### 2. 数据风险

如果过早引入真实点位，会把本次 UI 轮廓任务拉重。

因此一期要把专题入口配置与真实专题标注数据严格分开。

### 3. 表达风险

因为卡片只保留图片与标题，图片质量会直接影响观感。

所以后台自定义图片 URL 时，需要默认假定使用高质量横图素材。

## 当前结论

本次“中国自然地图”应先做成：

- 一个独立总地图包
- 一个前台专题大图流入口页
- 一个统一底图下的专题地图壳
- 一个后台可自定义图片与标题的专题入口管理页

而不是：

- 真实地理数据专题图
- 多个独立底图切换包
- 带复杂说明文案的内容管理系统
