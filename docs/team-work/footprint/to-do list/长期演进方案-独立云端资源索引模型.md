# 长期演进方案 - 独立云端资源索引模型

## 目标

- 将网盘挂载能力沉淀为长期基础设施
- 不让当前 `storage_files` 和 `OuterFrame` 结构绑定未来所有 Footprint 形态
- 支持后续扩展更多资源来源、更多展示形态、更多访问策略

## 适用前提

- Footprint 未来不止一种形态
- 云端资源不只是“当前页面补相册”
- 项目接受比“最小接入”更高的首期建模成本

## 核心结论

- 推荐建立独立的“云端资源索引层”
- `storage_files` 保留为当前本地上传与现有页面兼容层
- 新的 Footprint 资源应优先围绕 `cloud assets + footprint projections` 建模
- 当前 OuterFrame 只是一个 projection，不应定义资源底座

## 总体架构

```text
网盘 / AList / WebDAV / 本地挂载目录
        ↓
资源接入器（connector）
        ↓
cloud_asset_roots
cloud_assets
cloud_asset_sync_jobs
        ↓
footprint_asset_bindings
footprint_view_projections
        ↓
OuterFrame / 时间轴 / 相册墙 / 地图聚类 / 故事流
```

## 数据模型建议

### 1. `cloud_asset_roots`

表示某个用户的一块云端资源空间。

建议字段：

- `id`
- `user_id`
- `provider`
  - `alist` / `webdav` / `mount`
- `root_key`
  - 用户空间唯一标识
- `root_path`
- `display_name`
- `status`
  - `active / disabled / error`
- `sync_strategy`
  - `manual / scheduled / webhook`
- `last_synced_at`
- `created_at`
- `updated_at`

职责：

- 记录“这个用户接入了哪块云端空间”
- 支撑一个用户未来拥有多个根目录，而不是只有一个 `/user_{id}`

### 2. `cloud_assets`

表示实际文件资源，是长期稳定主表。

建议字段：

- `id`
- `root_id`
- `user_id`
- `provider_file_id`
  - 如果底层能提供稳定 ID，优先存
- `relative_path`
- `file_name`
- `file_ext`
- `mime_type`
- `size`
- `checksum`
- `etag`
- `version`
- `thumb_ref`
- `origin_ref`
- `captured_at`
- `modified_at`
- `last_seen_at`
- `deleted_at`
- `access_state`
  - `ok / expired / forbidden / missing`
- `metadata_json`

职责：

- 记录所有“用户可控文件”
- 保持对底层资源变化的可追踪性

### 3. `cloud_asset_sync_jobs`

表示同步与扫描任务。

建议字段：

- `id`
- `root_id`
- `job_type`
  - `full_scan / incremental / repair`
- `status`
  - `pending / running / success / failed`
- `cursor`
- `summary_json`
- `started_at`
- `finished_at`

职责：

- 支撑挂载目录扫描、断点续跑、失败修复

### 4. `footprint_asset_bindings`

将云端资源绑定到 Footprint 业务语义。

建议字段：

- `id`
- `asset_id`
- `user_id`
- `place_title`
- `place_id`
- `group_id`
- `binding_source`
  - `manual / folder_rule / filename_rule / imported / ai`
- `confidence`
- `is_cover_candidate`
- `visibility`
- `sort_order`
- `tags_json`
- `created_at`
- `updated_at`

职责：

- 管理“这张图属于哪个地点/分组/集合”
- 将文件资源与 Footprint 概念解耦

### 5. `footprint_view_projections`

专门承载“某种 Footprint 形态的展示字段”。

建议字段：

- `id`
- `binding_id`
- `user_id`
- `view_type`
  - `outer_frame / timeline / gallery_wall / cluster_map / story`
- `layout_key`
  - 同一形态下的布局版本或布局集合
- `x`
- `y`
- `z`
- `width`
- `height`
- `angle`
- `extra_json`
- `created_at`
- `updated_at`

职责：

- 不同视图各管各的布局数据
- 避免 `frame_x/frame_y` 变成全局资源字段

说明：

- 对于 `outer_frame`，`x/y` 就是当前 `frame_x/frame_y` 的演化形式
- 对于时间轴或故事流，`extra_json` 可存页序、章节等字段

## 为什么不直接继续扩展 `storage_files`

### 继续扩展的后果

- 本地上传、云端挂载、布局状态、地点绑定、访问控制都会堆进同一张表
- 将来加第二种、第三种 Footprint 形态时，表结构会不断被页面需求反向驱动
- 同步失败、远端删除、路径变更等云端特有状态会污染当前上传逻辑

### 独立建模的收益

- 云端资源生命周期和页面展示生命周期分离
- 可为不同 provider 定义不同同步策略
- 可同时支持多个视图形态而不挤压当前字段
- 后续如果需要资源审计、批量重建索引、失效检测，更容易实现

## 与现有系统的关系

### 当前 `storage_files`

建议定位：

- 保留为现有上传体系的主表
- 继续服务当前本地上传、文件权限、已有照片展示
- 不作为未来所有云端资源的唯一承载表

### 当前 AList 方案

建议定位：

- 作为 `provider=alist` 的首个 connector
- 负责提供目录列举、图片访问、签名链接、根路径隔离
- 不直接决定 Footprint 页面数据结构

### 当前 OuterFrame 页面

建议定位：

- 作为第一个 `view_type=outer_frame`
- 初期可通过投影接口读取混合数据：
  - `storage_files`
  - `cloud_assets + bindings + projections`
- 后续逐步切换到统一 projection API

## API 设计方向

### 资源层 API

- `GET /api/cloud/roots`
- `POST /api/cloud/roots`
- `POST /api/cloud/roots/{id}/sync`
- `GET /api/cloud/assets`
- `GET /api/cloud/assets/{id}`
- `GET /api/cloud/assets/{id}/file`

### 绑定层 API

- `POST /api/footprints/bindings`
- `PATCH /api/footprints/bindings/{id}`
- `DELETE /api/footprints/bindings/{id}`

### 投影层 API

- `GET /api/footprints/views/{viewType}`
- `PATCH /api/footprints/views/{viewType}/projection`
- `POST /api/footprints/views/{viewType}/rebuild`

说明：

- 页面不再自己推断目录结构
- 页面只请求某个 view 的成品数据

## 用户隔离策略

### 资源接入阶段

- 每个 `root` 必须显式绑定 `user_id`
- 同步器只能在该 `root` 范围内扫描
- 所有 provider 路径都先规范化为相对路径，不保存业务侧可直接拼接的危险绝对路径

### 资源访问阶段

- 外部文件访问统一经过服务端文件网关
- 由服务端校验：
  - 当前登录用户
  - 资源归属的 `user_id`
  - 资源状态是否可访问
- 对前端返回短时有效链接或中转流

### 投影读取阶段

- 任何 projection 查询都必须按 `user_id` 限定
- 管理员查看他人数据，继续沿用当前 view token / admin 授权思路

## 迁移策略

### Phase 1

- 保持当前 `storage_files` 路径不动
- 引入 `cloud_asset_roots`、`cloud_assets`
- 跑通 AList 资源同步

### Phase 2

- 引入 `footprint_asset_bindings`
- 支持手动绑定地点、封面候选、分组映射

### Phase 3

- 引入 `footprint_view_projections`
- 让 OuterFrame 视图开始读取 projection 数据

### Phase 4

- 新增其他 Footprint 形态
- 各形态只增加新 projection，不回写到底层资源表

### Phase 5

- 评估是否逐步弱化 `storage_files` 在 Footprint 中的中心地位
- 但不强行迁移已有本地上传数据，避免高风险重构

## 主要风险

### 首期复杂度上升

- 表更多
- API 分层更多
- 同步链路更长

缓解：

- 先只实现 `alist` connector
- 先只支持图片资源
- 先只落一个 `outer_frame` projection

### 双轨数据并存

- `storage_files` 与 `cloud_assets` 会并行一段时间

缓解：

- 明确“上传链路”和“挂载链路”各自归属
- 页面通过统一 DTO 聚合，避免前端知道两种底层来源

### 路径与资源 ID 不稳定

- 某些 provider 只提供路径，不提供稳定 file id

缓解：

- 优先存 provider file id
- 没有时用 `root_id + relative_path + checksum` 做弱稳定标识

## 推荐决策

- 如果项目确认 Footprint 会持续多形态扩展，长期应走“独立云端资源索引模型”
- 当前 OuterFrame 不应成为云端资源系统的主模型
- `storage_files` 继续服务现有功能，但不再承接未来所有云端资源演进责任

## 下一步可继续细化

- `cloud_assets` 与 `storage_files` 的统一返回 DTO 方案
- `outer_frame` projection 的最小字段定义
- AList connector 的同步策略与失败恢复策略
- 地点绑定规则的优先级
