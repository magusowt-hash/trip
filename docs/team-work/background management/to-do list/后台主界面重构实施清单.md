# 后台主界面重构实施清单

## 对应方案

- 本清单对应 `后台管理页视图逻辑优化.md` 中确认采用的方案 B
- 目标是重构后台主界面入口与导航结构
- 本轮只处理入口视图与 layout，不改各子页面内部业务逻辑

## 实施目标

- 为后台管理页建立统一侧边栏导航
- 按 `系统管理`、`用户管理` 两个一级分组重组入口
- 让首页与侧边栏共用同一份导航配置，消除重复维护
- 保留后台首页的看板能力，同时弱化“全部入口平铺”的旧结构
- 同步优化后台主界面的 CSS 结构、视觉层级和响应式表现

## 第一阶段：统一导航配置

### 任务

- 新建后台导航配置源
- 抽离现有首页 `quickNav` 和 layout 中的 `navItems`
- 为每个入口统一定义：
  - `path`
  - `label`
  - `icon`
  - `description`
  - `group`
  - `order`

### 产出

- 单一导航配置文件
- 首页和 layout 改为消费统一配置

### 验收点

- 后台入口只维护一份配置
- 首页和侧边栏展示内容一致
- 面包屑和当前页标题不再依赖重复数组定义

## 第二阶段：重构后台 layout

### 任务

- 在 `src/app/management/layout.tsx` 中引入左侧导航栏
- 调整整体容器为：
  - 左侧导航
  - 右侧主内容区
- 保留顶部区域，但职责收敛为：
  - 当前页标题
  - 面包屑
  - 退出登录
- 同步整理 layout 相关 CSS，避免继续把大量壳层样式塞在内联 `style` 中

### 侧边栏要求

- 顶部保留“管理后台”入口
- 一级结构固定为：
  - 看板
  - 系统管理
  - 用户管理
- 二级入口直接列在对应分组下
- 当前页所在入口需要高亮
- 当前页所在分组要有明确视觉归属

### 验收点

- 所有后台页面进入后都能看到一致的左侧导航
- 从任意后台子页都能直接切换到同组或跨组页面
- 当前页高亮正确
- 登录页不受新 layout 导航影响
- 侧边栏、顶部栏、内容区之间层级清晰，视觉结构稳定

## 第三阶段：重构后台首页

### 任务

- 保留当前数据概览与趋势图能力
- 去掉旧的“所有入口平铺”
- 改为按分组展示首页入口摘要
- 重新整理首页 CSS，使看板信息、趋势图和分组导航形成清晰主次

### 首页建议结构

- 页面标题区
- 数据概览区
- 趋势图区
- 系统管理摘要区
- 用户管理摘要区

### 首页摘要区要求

- 每个分组展示分组说明
- 每个分组展示若干入口卡片
- 卡片信息与侧边栏配置保持一致
- 首页卡片承担“概览”和“快速进入”职责，不替代完整导航

### 验收点

- 首页不再是无分类入口墙
- 分组结构与侧边栏一致
- 首页视觉层次明显优于现状
- 首页卡片与侧边栏在风格上保持一致，不再像两套独立界面

## 第四阶段：导航联动校正

### 任务

- 校正面包屑标题来源
- 校正首页、子页、分组之间的跳转逻辑
- 校验所有管理页是否都落入正确分组
- 校验新增 CSS 不会压坏现有管理页的表格和内容宽度

### 验收点

- 每个后台页面都能正确识别所属导航项
- 面包屑名称与侧边栏文案一致
- 不存在漏挂载、错分组或高亮错误

## 风险点

- 当前 `layout.tsx` 已承担鉴权与跳转逻辑，重构时要避免影响登录态判断
- 首页和 layout 目前分别维护导航数据，迁移时要确保文案与路径完全对齐
- 如果页面宽度和表格布局比较紧，新增侧边栏后主内容宽度可能不足，需要同步检查各管理页展示
- `maps`、`users/[id]`、`lists/[id]` 这类带层级的页面需要额外验证高亮和面包屑匹配逻辑
- 现有后台页面样式主要以内联方式组织，本轮 CSS 调整如果边界不清，容易影响未改造页面

## CSS 专项要求

### 布局

- 后台整体改为稳定的双栏结构
- 侧边栏宽度固定，主内容区自适应
- 页面顶部区域应与内容区对齐，不要出现多套不一致的左右边距

### 导航

- 分组标题、普通导航项、当前导航项三者要有明确层级
- 当前项不能只依赖文字颜色，需要有稳定高亮背景或边界强调
- 悬停状态应轻量，不要破坏后台工具型界面的稳定感

### 首页

- 数据概览、趋势图、分组入口三部分要有清晰节奏
- 分组卡片内部要区分分组标题、说明、入口项
- 入口项不应继续使用完全等权的满屏平铺方式

### 响应式

- 桌面端优先保证双栏结构完整
- 中等宽度下需要控制内容压缩，避免表格区域不可用
- 登录页继续保持独立布局，不继承后台主壳层样式

## 验证清单

- 打开 `/management`，确认首页分为 `系统管理`、`用户管理`
- 打开 `/management/users`，确认落在 `用户管理`
- 打开 `/management/maps`，确认落在 `系统管理`
- 打开 `/management/embed-logs`，确认落在 `系统管理`
- 打开 `/management/footprints`，确认落在 `用户管理`
- 从侧边栏跨分组跳转，确认高亮和标题同步变化
- 登录页 `/management/login` 不显示侧边栏和后台主壳层
- 在常见后台页面检查主内容宽度，确认表格和表单未被新侧边栏压坏
- 检查首页、侧边栏、顶部栏样式是否属于同一套视觉体系

## 实施建议

- 先做导航配置抽离，再做 layout
- 不建议先改首页，再补侧边栏；否则会产生两套过渡结构
- 本轮应优先解决“结构统一”，不要把子页面内部重构混进同一任务

## 结论

- 当前实施应按“配置统一 -> layout 重构 -> 首页重构 -> 联动校正”推进
- 只要导航配置源收敛成功，后续后台入口扩展成本会明显下降
