# 第一阶段数据模型草案

## 目标

- 为第一阶段“手动同步网盘 -> 目录名匹配地点 -> 已绑定展示 / 未绑定提示”提供可实现的数据模型
- 保持云端资源域独立，不与现有 `storage_files` 合表
- 为后续多形态 Footprint 保留 projection 扩展位

## 适用范围

本草案仅覆盖第一阶段所需最小模型：

- 用户云端根目录
- 云端一级文件夹
- 云端图片资源
- 目录/资源到 Footprint 地点的绑定
- `outer_frame` 布局 projection

不覆盖：

- 定时任务调度系统
- 全站标准地点库
- 多 provider 高级差异字段
- 复杂审核流

## 表清单

- `cloud_asset_roots`
- `cloud_asset_folders`
- `cloud_assets`
- `footprint_cloud_bindings`
- `footprint_view_projections`
- `cloud_asset_sync_logs`

## 1. `cloud_asset_roots`

表示用户的一块云端根目录空间。

### 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | bigint PK | 主键 |
| `user_id` | bigint not null | 所属用户 |
| `provider` | varchar(32) not null | `alist` / `webdav` / `mount` |
| `root_key` | varchar(128) not null | 根空间唯一标识 |
| `root_path` | varchar(512) not null | 逻辑根路径 |
| `display_name` | varchar(128) not null | 展示名称 |
| `status` | varchar(32) not null | `active / disabled / error` |
| `last_synced_at` | datetime null | 最近同步时间 |
| `created_at` | datetime not null | 创建时间 |
| `updated_at` | datetime not null | 更新时间 |

### 约束

- 唯一键：`uniq_user_root_key (user_id, root_key)`

### 索引

- `idx_cloud_asset_roots_user_id (user_id)`
- `idx_cloud_asset_roots_status (status)`

### 说明

- 第一阶段一个用户原则上可先只用一条 root
- 但表结构不限制未来扩展多个 root

## 2. `cloud_asset_folders`

表示根目录第一层文件夹。

### 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | bigint PK | 主键 |
| `root_id` | bigint not null | 所属根目录 |
| `user_id` | bigint not null | 所属用户 |
| `folder_name` | varchar(255) not null | 一级文件夹名 |
| `relative_path` | varchar(512) not null | 相对根目录路径 |
| `match_status` | varchar(32) not null | `matched / unbound` |
| `matched_place_title` | varchar(255) null | 命中的地点名 |
| `asset_count` | int not null default 0 | 文件夹内图片数量 |
| `sample_asset_id` | bigint null | 用于提示的样例图 |
| `last_scanned_at` | datetime null | 最近扫描时间 |
| `created_at` | datetime not null | 创建时间 |
| `updated_at` | datetime not null | 更新时间 |

### 约束

- 唯一键：`uniq_folder_root_path (root_id, relative_path)`

### 索引

- `idx_cloud_asset_folders_user_id (user_id)`
- `idx_cloud_asset_folders_root_id (root_id)`
- `idx_cloud_asset_folders_match_status (match_status)`
- `idx_cloud_asset_folders_matched_place_title (matched_place_title)`

### 说明

- 第一阶段只扫描一级目录，因此 `folder_name` 是主要匹配字段
- `matched_place_title` 先直接存字符串，不强依赖标准地点 ID

## 3. `cloud_assets`

表示具体云端图片资源。

### 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | bigint PK | 主键 |
| `root_id` | bigint not null | 所属根目录 |
| `folder_id` | bigint not null | 所属一级文件夹 |
| `user_id` | bigint not null | 所属用户 |
| `provider_file_id` | varchar(255) null | provider 稳定文件 ID |
| `relative_path` | varchar(1024) not null | 相对根目录文件路径 |
| `file_name` | varchar(255) not null | 文件名 |
| `file_ext` | varchar(32) null | 扩展名 |
| `mime_type` | varchar(128) null | MIME 类型 |
| `size` | bigint null | 文件大小 |
| `width` | int null | 图片宽 |
| `height` | int null | 图片高 |
| `checksum` | varchar(128) null | 文件摘要 |
| `etag` | varchar(255) null | provider 版本标识 |
| `thumb_ref` | varchar(1024) null | 缩略图引用 |
| `origin_ref` | varchar(1024) null | 原图引用 |
| `captured_at` | datetime null | 拍摄时间 |
| `access_state` | varchar(32) not null | `ok / expired / forbidden / missing` |
| `sync_state` | varchar(32) not null | `ready / skipped / error` |
| `is_image` | tinyint(1) not null default 1 | 是否图片 |
| `deleted_at` | datetime null | 远端删除时间 |
| `last_seen_at` | datetime null | 最近在网盘扫描中可见时间 |
| `created_at` | datetime not null | 创建时间 |
| `updated_at` | datetime not null | 更新时间 |

### 约束

- 唯一键：`uniq_asset_root_path (root_id, relative_path)`

### 索引

- `idx_cloud_assets_user_id (user_id)`
- `idx_cloud_assets_root_id (root_id)`
- `idx_cloud_assets_folder_id (folder_id)`
- `idx_cloud_assets_access_state (access_state)`
- `idx_cloud_assets_sync_state (sync_state)`
- `idx_cloud_assets_last_seen_at (last_seen_at)`

### 说明

- 第一阶段只入库图片，因此 `is_image` 理论上恒为 1，但保留字段利于以后扩展
- `origin_ref` / `thumb_ref` 只存服务端可解释引用，不直接假设前端拿到的是永久外链

## 4. `footprint_cloud_bindings`

表示云端目录或图片与 Footprint 地点的绑定关系。

### 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | bigint PK | 主键 |
| `user_id` | bigint not null | 所属用户 |
| `folder_id` | bigint not null | 绑定来源目录 |
| `asset_id` | bigint not null | 绑定图片 |
| `place_title` | varchar(255) not null | 命中的足迹地点名 |
| `group_id` | bigint null | 若后续接足迹分组可填 |
| `binding_source` | varchar(32) not null | 第一阶段固定为 `folder_name_exact` |
| `binding_status` | varchar(32) not null | `bound / unbound` |
| `visibility` | varchar(32) not null | `private / friends / public`，第一阶段可默认 private |
| `created_at` | datetime not null | 创建时间 |
| `updated_at` | datetime not null | 更新时间 |

### 约束

- 唯一键：`uniq_binding_asset (asset_id)`

### 索引

- `idx_footprint_cloud_bindings_user_id (user_id)`
- `idx_footprint_cloud_bindings_folder_id (folder_id)`
- `idx_footprint_cloud_bindings_place_title (place_title)`
- `idx_footprint_cloud_bindings_binding_status (binding_status)`

### 说明

- 第一阶段按“目录命中后该目录下所有图片自动归属该地点”，因此每个 `asset` 只应有一条有效绑定
- 未匹配资源可不写 binding，或写 `binding_status=unbound`
- 建议第一阶段优先采用“不匹配则不写 binding”，由 `cloud_asset_folders.match_status` 与资源归属目录来表示未匹配状态，避免重复语义

## 5. `footprint_view_projections`

表示某个资源在具体 Footprint 视图中的布局信息。

### 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | bigint PK | 主键 |
| `user_id` | bigint not null | 所属用户 |
| `asset_id` | bigint not null | 对应图片资源 |
| `view_type` | varchar(32) not null | 第一阶段固定 `outer_frame` |
| `layout_key` | varchar(64) not null | 布局版本或集合标识，第一阶段可默认 `default` |
| `x` | double null | 视图 x 坐标 |
| `y` | double null | 视图 y 坐标 |
| `z` | double null | 层级或深度，可空 |
| `width` | double null | 展示宽度，可空 |
| `height` | double null | 展示高度，可空 |
| `angle` | double null | 旋转角度，可空 |
| `extra_json` | json null | 扩展字段 |
| `created_at` | datetime not null | 创建时间 |
| `updated_at` | datetime not null | 更新时间 |

### 约束

- 唯一键：`uniq_projection_asset_view_layout (asset_id, view_type, layout_key)`

### 索引

- `idx_footprint_view_projections_user_id (user_id)`
- `idx_footprint_view_projections_asset_id (asset_id)`
- `idx_footprint_view_projections_view_type (view_type)`

### 说明

- 当前 `frame_x/frame_y` 在这一层表达为 `x/y`
- 未来多形态视图只需新增 `view_type`，不需要改资源主表

## 6. `cloud_asset_sync_logs`

表示每次手动同步的摘要。

### 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | bigint PK | 主键 |
| `root_id` | bigint not null | 同步根目录 |
| `user_id` | bigint not null | 所属用户 |
| `triggered_by` | varchar(32) not null | `user / admin` |
| `sync_status` | varchar(32) not null | `success / failed` |
| `scanned_folder_count` | int not null default 0 | 扫描目录数 |
| `imported_asset_count` | int not null default 0 | 新入库资源数 |
| `skipped_asset_count` | int not null default 0 | 跳过资源数 |
| `matched_folder_count` | int not null default 0 | 命中目录数 |
| `unbound_folder_count` | int not null default 0 | 未命中目录数 |
| `error_code` | varchar(64) null | 错误码 |
| `error_message` | varchar(255) null | 错误摘要 |
| `started_at` | datetime not null | 开始时间 |
| `finished_at` | datetime null | 结束时间 |
| `created_at` | datetime not null | 创建时间 |

### 索引

- `idx_cloud_asset_sync_logs_user_id (user_id)`
- `idx_cloud_asset_sync_logs_root_id (root_id)`
- `idx_cloud_asset_sync_logs_started_at (started_at)`

### 说明

- 第一阶段没有自动调度，但同步日志仍然必要
- 便于用户端提示最近一次同步结果，也便于管理端排查失败

## 推荐外键关系

```text
cloud_asset_roots.id -> cloud_asset_folders.root_id
cloud_asset_roots.id -> cloud_assets.root_id
cloud_asset_folders.id -> cloud_assets.folder_id
cloud_asset_folders.id -> footprint_cloud_bindings.folder_id
cloud_assets.id -> footprint_cloud_bindings.asset_id
cloud_assets.id -> footprint_view_projections.asset_id
cloud_asset_roots.id -> cloud_asset_sync_logs.root_id
```

说明：

- `user_id` 仍建议冗余存储在各表中，便于快速鉴权和查询

## 第一阶段状态枚举建议

### `cloud_asset_roots.status`

- `active`
- `disabled`
- `error`

### `cloud_asset_folders.match_status`

- `matched`
- `unbound`

### `cloud_assets.access_state`

- `ok`
- `expired`
- `forbidden`
- `missing`

### `cloud_assets.sync_state`

- `ready`
- `skipped`
- `error`

### `footprint_cloud_bindings.binding_source`

- `folder_name_exact`

### `footprint_cloud_bindings.binding_status`

- `bound`

说明：

- 第一阶段如果未匹配则建议不创建 binding 记录，因此 `binding_status` 实际上可以只有 `bound`
- 若后续希望记录解绑历史，再扩充为 `unbound / removed`

## 查询模式建议

### 当前页面读取已绑定云端图片

查询链路：

1. 按 `user_id` 查 `footprint_cloud_bindings`
2. join `cloud_assets`
3. left join `footprint_view_projections`
4. 按 `place_title` 聚合

### 用户端读取未匹配提示

查询链路：

1. 按 `user_id` 查 `cloud_asset_folders`
2. 过滤 `match_status = 'unbound'`
3. 取 `asset_count + sample_asset_id`
4. join 一张样例资源图用于提示

### 同步时去重

去重优先级建议：

1. `root_id + relative_path`
2. 若 provider 提供稳定 ID，可辅助校验 `provider_file_id`
3. 如需更稳，可再比对 `etag` 或 `checksum`

## 与现有 `storage_files` 的边界

- `storage_files`
  - 继续承接本地上传
  - 继续承接现有文件 API 能力
- 新模型
  - 只承接网盘同步资源
  - 不直接写入 `storage_files`

说明：

- 第一阶段页面层可以在服务端 DTO 聚合时混合本地和云端
- 但数据层保持双轨，避免过早耦合

## 索引与性能建议

- 所有主查询都必须可按 `user_id` 起手
- `relative_path` 作为幂等键必须建唯一索引
- `match_status`、`access_state` 必须有单列索引，方便提示和排障
- `asset_id + view_type + layout_key` 的唯一键必须有，避免重复 projection

## 第一阶段简化建议

如果要进一步降低复杂度，可做以下收缩：

- `group_id` 先不落库，统一为空
- `visibility` 先不提供编辑入口，默认 `private`
- `width/height/captured_at/checksum` 可先尽力写，获取失败允许为空
- `extra_json` 先保留但不使用

## 推荐 SQL 方向

推荐用与现有项目一致的 Drizzle + MySQL 风格实现，新增：

- `src/db/schema.cloudAssets.ts`
- 在 `src/db/schema.ts` 中统一导出
- 独立迁移文件创建以上表
