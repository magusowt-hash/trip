# 地图模块化实施方案

## 方案目标

- 地图模块按“地图类型”拆分，而不是按接口或页面拆分
- 当前至少包含：
  - 普通地图
  - 铁路地图
  - 未来新增专题地图
- 后台统一保留一个“地图管理”入口
- 各地图的后台管理界面、接口、配置、数据处理逻辑来自各自地图包内部
- 能共享的能力沉淀到公共区，避免重复造轮子
- 隔离的目的以“便于管理、便于扩展、便于替换”为主，不追求机械式绝对隔离

## 当前实际结构分析

### 1. 前台地图页当前是单页混装

- 当前前台地图入口是 `src/app/(shell)/maps/page.tsx`
- 普通地图与铁路地图在同一个页面中通过 `activeTab` 切换
- 普通地图依赖：
  - `PlanMap`
  - `/api/maps/search`
  - `/api/maps/selection`
  - `/api/maps/favorites`
  - `/api/maps/footprints`
- 铁路地图依赖：
  - `PlanMap`
  - `RailCanvas`
  - `/data/railways.json`
  - `/data/stations.json`
  - `/api/public/rail-settings`
  - `/api/public/station-overrides`

结论：
- 当前是“一个入口页内分支化承载多种地图”
- 新增地图类型会继续往这个页面追加 `tab + 状态 + 数据请求 + 视图分支`
- 这会导致页面复杂度、状态耦合、测试成本持续上升

### 2. 后台地图管理当前是公共目录直连

- 当前后台总入口是 `src/app/management/maps/page.tsx`
- 当前铁路管理页是 `src/app/management/maps/rail/page.tsx`
- 当前铁路后台接口是 `src/app/api/admin/maps/rail/settings/route.ts`

结论：
- 当前后台“地图管理”虽然已有统一入口，但子地图管理界面仍然直接写在公共目录
- 这种结构在地图种类少时可用，但随着地图包增加，会变成公共目录不断膨胀

### 3. 数据层当前呈现“共享基础地图 + 铁路专属数据”趋势

- 普通地图已有：
  - `map_pois`
  - `user_map_favorites`
  - `user_map_footprints`
- 铁路地图已有：
  - `rail_map_settings`
- 铁路静态数据还依赖本地 JSON 文件

结论：
- 当前仓库已经天然表现出“共享少量基础能力 + 各地图拥有专属数据”的结构
- 因此不适合强行统一所有地图表结构
- 更合适的是建立：
  - 通用公共区
  - 各地图包专属数据区

## 模块化设计原则

### 1. 按地图类型拆包

- `standard-map` 负责普通地图
- `rail-map` 负责铁路地图
- 后续新增地图直接新增包，例如：
  - `metro-map`
  - `scenic-map`
  - `hiking-map`

### 2. 公共区只放“每张地图都会有”的能力

- 地图包注册协议
- 地图管理后台聚合逻辑
- 公共地图容器页
- 公共鉴权能力
- 公共错误处理约定
- 公共类型协议
- 公共配置读取能力

不应放入公共区的内容：
- 某种地图专属数据结构
- 某种地图专属后台表单
- 某种地图专属查询逻辑
- 某种地图专属渲染算法

### 3. 地图包自带完整能力

- 每个地图包内部自行维护：
  - 前台展示实现
  - 后台管理页面
  - API 处理
  - 服务逻辑
  - 类型定义
  - 配置读取
  - 专属数据访问层

### 4. 后台统一入口，子能力由包注册

- 后台保留统一入口 `/management/maps`
- 不把每个地图都做成后台一级菜单
- 地图管理首页负责读取地图包注册信息并展示卡片
- 点击后进入对应地图包的管理页

## 后台页面来源方式分析

这里有两个可选方向。

### 方案 A：地图包导出后台注册对象，由公共后台挂载

做法：
- 每个地图包导出：
  - `adminMeta`
  - `adminEntry`
  - `adminRoutes`
  - `permissionKey`
- 公共后台 `/management/maps` 根据注册对象生成入口卡片
- 公共动态路由页根据 `slug` 找到对应地图包并渲染其后台页面

优点：
- 最符合“后台统一入口，页面来源于地图包内部”
- 新增地图主要是“新增一个地图包并注册”
- 公共后台目录不会持续膨胀
- 管理入口、描述、权限、页面来源都可以统一收口
- 更适合未来新增很多地图类型

缺点：
- 需要建立一层注册协议
- 页面挂载方式比直接写死路由稍复杂
- 某些依赖 Next 文件路由能力的页面，需要额外设计适配层

### 方案 B：每个地图包自己生成 `app/management/maps/...` 路由文件

做法：
- 地图包自己提供页面实现
- 但最终仍在 `src/app/management/maps/...` 下放实际页面文件

优点：
- 对 Next App Router 更直观
- 初期开发成本低
- 不需要动态挂载协议也能工作

缺点：
- 表面上来源于地图包，实际上公共后台目录仍会越来越大
- 新增地图仍需要改公共目录
- 不利于长期维护“地图包自治”
- 容易回到现在铁路地图这种公共目录膨胀模式

### 结论

优先推荐方案 A。

原因：
- 你的核心目标不是“把文件换个地方放”，而是“每个地图包自带后台能力，公共后台只负责聚合管理”
- 方案 A 更符合这个目标
- 方案 B 只是在形式上拆文件，长期仍会把公共路由目录变成地图包集散地

## 推荐后台挂载方式

建议采用下面的结构：

```text
src/app/management/maps/page.tsx
src/app/management/maps/[packageSlug]/page.tsx
```

当前确认的 URL 风格：

- 普通地图：`/management/maps/standard`
- 铁路地图：`/management/maps/rail`
- 地图包目录名仍保持：
  - `standard-map`
  - `rail-map`

职责划分：

- `src/app/management/maps/page.tsx`
  - 读取已注册地图包
  - 渲染地图管理首页卡片

- `src/app/management/maps/[packageSlug]/page.tsx`
  - 根据 `packageSlug` 查找地图包
  - 渲染该地图包导出的后台主页组件

每个地图包导出：

```ts
export const mapPackage = {
  slug: 'rail',
  name: '中国铁路地图',
  admin: {
    description: '站点显示参数、覆盖管理',
    renderPage: RailMapAdminPage,
  },
};
```

这样可以做到：
- 后台入口统一
- 后台页面实际来源于地图包内部
- 新增地图只注册，不扩散公共后台代码

## 前台地图页推荐改造方式

当前不建议继续在 `src/app/(shell)/maps/page.tsx` 中追加更多 `tab` 分支。

建议改为“公共地图容器页 + 地图包视图注册”：

```text
src/modules/maps/core/ui/maps-shell/
src/modules/maps/packages/standard-map/ui/
src/modules/maps/packages/rail-map/ui/
src/app/(shell)/maps/page.tsx
```

职责：

- `src/app/(shell)/maps/page.tsx`
  - 仅作为页面入口
  - 调用公共地图容器

- 公共地图容器
  - 读取已注册地图包
  - 控制当前激活地图类型
  - 渲染地图类型切换 UI

- 各地图包 UI
  - 提供自己的地图主视图
  - 提供自己的侧边面板
  - 提供自己的数据加载逻辑

这样普通地图与铁路地图不再共处一个大组件，而是：
- 一个公共壳
- 两个独立包实现

## 数据层策略分析

你的方向是“能共享则共享”。结合当前实际，推荐采用：

### 1. 共享“基础模型”，不强行共享“业务专属模型”

可以共享：
- 通用地图包注册表述
- 通用配置读取约定
- 通用用户鉴权
- 通用后台鉴权
- 通用公共 POI 能力（如果未来确实多地图复用）
- 通用地图元信息

不建议强行共享：
- 铁路地图设置表与普通地图 POI 表
- 铁路专属站点覆盖逻辑与普通地图收藏/足迹逻辑
- 各地图专属渲染参数表

### 2. 数据表设计原则

- 如果是“所有地图都会有”的数据，再抽公共表
- 如果是“某一种地图明显专属”的数据，直接独立表
- 不为了表面统一，把差异很大的数据硬塞进一张超级大表

### 3. 对当前仓库的实际建议

- `map_pois`、`user_map_favorites`、`user_map_footprints`
  - 先视为普通地图包专属
  - 除非后续新地图也明确复用 POI 收藏/足迹模型，再考虑上收公共区

- `rail_map_settings`
  - 明确保留在铁路地图包专属域

- `stations.json`、`railways.json`
  - 视为铁路地图包专属数据源

结论：
- 当前阶段不建议为了“将来也许共享”而提前过度抽象数据库
- 先保留现有数据归属，只抽真正已经稳定的公共能力

## 迁移策略分析

### 不建议一次性大迁移

原因：
- 当前前台地图页、后台铁路管理页、铁路 API、普通地图 API 都已在线形成依赖关系
- 一次性整体迁移会把前台、后台、接口、数据访问同时打散，回归风险高

### 推荐分阶段迁移

#### 第一阶段：建壳，不改行为

- 建立 `src/modules/maps/core`
- 建立 `src/modules/maps/packages/standard-map`
- 建立 `src/modules/maps/packages/rail-map`
- 把“地图包注册协议”搭起来
- 把现有后台 `maps` 首页改为读取注册信息
- 不立即大量改动数据逻辑

目标：
- 先让目录结构和模块边界成立
- 行为与现状尽量保持一致

#### 第二阶段：迁后台

- 把铁路地图后台页面实现迁入 `rail-map/admin`
- 公共后台通过 `[packageSlug]` 挂载地图包后台页
- 保留旧路由一段时间做兼容跳转或薄适配

目标：
- 完成“后台页面来源于地图包内部”

#### 第三阶段：迁前台地图容器

- 抽出公共地图容器页
- 普通地图 UI 迁入 `standard-map/ui`
- 铁路地图 UI 迁入 `rail-map/ui`
- 让 `/maps` 从“大页面分支逻辑”变为“公共壳 + 地图包视图”

目标：
- 解除前台不同地图类型之间的组件耦合

#### 第四阶段：迁接口与服务

- 普通地图 API 逻辑迁入 `standard-map/api`
- 铁路地图接口逻辑迁入 `rail-map/api`
- Route 文件只做 HTTP 适配和调用转发

目标：
- 让公共 `app/api` 目录只承担路由入口职责

#### 第五阶段：收口公共区

- 对已经明确多地图复用的能力再上收公共区
- 不复用的不抽象

目标：
- 防止公共区变成新的杂物间

## 第一阶段建议产出

### 代码结构层

- 建立地图模块主目录
- 建立地图包注册协议
- 建立标准地图包骨架
- 建立铁路地图包骨架
- 建立后台地图管理首页的注册式渲染
- 确认目录名使用 `standard-map` / `rail-map`
- 确认后台 slug 使用 `standard` / `rail`

### 文档层

- 地图包命名规范
- 地图包注册规范
- 后台挂载规范
- 公共区准入规则

## 风险点

- 如果公共区定义过宽，会很快退化成“什么都能往里放”的新耦合点
- 如果动态挂载协议设计过重，会拖慢开发速度
- 如果在第一阶段就强行重构所有 API 和数据访问，回归风险会明显增大

## 最终建议

- 以“地图类型”作为一级模块单位
- 后台保留统一的“地图管理”入口
- 后台管理页面由各地图包内部导出，公共后台负责注册式挂载
- 数据层坚持“能共享则共享，但只共享已经稳定通用的部分”
- 迁移方式采用分阶段推进，先建壳，再迁后台，再迁前台，再迁接口
