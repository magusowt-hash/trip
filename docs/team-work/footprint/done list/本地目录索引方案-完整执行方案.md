# 本地目录索引方案 - 完整执行方案

## 一、目标

将 `/test-css` 已验证通过的“本地主文件夹索引 + 文本持久化 + 排序恢复 + 差异提示”迁移为 `footprint` 模块内的正式第一阶段方案。

本方案只覆盖：

- 用户在 Footprint 页面中选择本地主文件夹
- 读取并展示该目录图片
- 根据第一层子文件夹映射地点
- 保存组内排序
- 二次进入后重新选择同名主文件夹并恢复排序
- 提示目录内容差异

本方案不覆盖：

- 原始图片上传
- 自由布局坐标
- 数据库存储
- 云端挂载 / AList / WebDAV

## 二、整体架构

### 前端职责

- 触发目录选择
- 读取 `FileList`
- 解析主文件夹名、子文件夹名、图片相对路径
- 生成预览图
- 按地点目录分组展示
- 支持组内排序调整
- 调用服务端接口读取 / 保存文本记录
- 展示差异结果

### 服务端职责

- 为每个主文件夹名称保存一份文本记录
- 返回对应主文件夹的历史排序与摘要
- 维护多主文件夹记录列表
- 不保存原始图片二进制

### 文本记录职责

- 记录主文件夹名称
- 记录图片摘要
- 记录排序位置
- 记录最近保存时间
- 记录轻量页面设置

## 三、正式页面接入策略

### 入口位置

在 `footprint` 用户页中增加一个新的用户动作入口：

- `选择本地主文件夹`

入口形态建议：

- 放在当前图片相关操作区域
- 与“上传照片”语义分开
- 文案必须明确这是“本地目录索引”，不是上传

### 页面表现

用户选择目录后：

1. 显示主文件夹名称
2. 显示已命中的历史记录状态
3. 按第一层子文件夹分组显示图片
4. 显示差异统计
5. 支持排序调整并保存

## 四、数据结构

### 服务端文本记录结构

```json
{
  "version": 1,
  "sessions": [
    {
      "version": 1,
      "rootName": "JapanTrip",
      "savedAt": "2026-05-21T12:00:00.000Z",
      "files": [
        {
          "relativePath": "JapanTrip/东京/a.jpg",
          "folderName": "东京",
          "name": "a.jpg",
          "size": 123456,
          "lastModified": 1710000000000,
          "sortOrder": 0
        }
      ]
    }
  ]
}
```

### 前端运行态结构

- `rootName`
- `images[]`
  - `relativePath`
  - `folderName`
  - `name`
  - `size`
  - `lastModified`
  - `url`
  - `sortOrder`
- `diffSummary`
- `persistedSession`

## 五、接口方案

### 1. 读取主文件夹记录

`GET /api/footprints/local-index/session?rootName={rootName}`

返回：

- 该主文件夹名称对应的文本记录
- 已知主文件夹名称列表

### 2. 保存主文件夹记录

`POST /api/footprints/local-index/session`

请求体：

- `rootName`
- `files[]`
- `viewSettings`

返回：

- 保存后的最新记录
- 已知主文件夹名称列表

## 六、前端执行流程

### 首次进入页面

1. 页面不主动读取目录内容。
2. 页面提示用户选择本地主文件夹。
3. 如有需要，可先读取“已知主文件夹名称列表”用于提示。

### 用户选择主文件夹

1. 通过文件选择器拿到 `FileList`。
2. 从 `webkitRelativePath` 中解析主文件夹名称。
3. 调用 `GET session?rootName=...` 读取历史记录。
4. 过滤非图片文件。
5. 将图片按第一层子文件夹分组。
6. 若历史记录存在，则按 `relativePath -> sortOrder` 复用排序。
7. 计算差异：
   - 新增
   - 缺失
   - 已变化
8. 展示页面。

### 用户调整排序

1. 在组内进行上移 / 下移。
2. 前端只更新当前会话中的 `sortOrder`。
3. 未点击保存前，不写服务端文本。

### 用户保存排序

1. 将当前图片列表转为文本记录结构。
2. 调用 `POST session`。
3. 成功后刷新当前 `persistedSession`。
4. 页面显示“已保存”时间。

### 用户二次进入

1. 再次选择同名主文件夹。
2. 页面先读该主文件夹记录。
3. 若相对路径命中，则恢复旧排序。
4. 若目录有变动，则提示差异，但保留可命中的排序结果。

## 七、匹配与差异规则

### 地点目录匹配规则

- 只认第一层子文件夹
- 目录名标准化仅做轻量处理：
  - `trim`
  - 压缩连续空格
- 与地点名完全一致才算命中

### 文件差异规则

#### 无差异

- `relativePath` 存在于历史记录中
- 且 `size`、`lastModified`、`name`、`folderName` 一致

#### 新增文件

- 当前扫描有
- 历史记录中无同 `relativePath`

#### 缺失文件

- 历史记录中有
- 当前扫描中无同 `relativePath`

#### 已变化文件

- `relativePath` 相同
- 但 `size`、`lastModified`、`name`、`folderName` 任一变化

## 八、与现有 Footprint 的边界

### 不与 `storage_files` 混用

- `storage_files` 继续只处理真正上传图片
- 本地目录索引图片不进入 `storage_files`

### 不改现有外框设置逻辑

- `footprint settings` 仍走现有接口
- 本地目录索引只额外保存轻量排序与索引设置

### 不接管理端

- 第一阶段只做用户侧
- 管理端不接本地目录索引查看能力

## 九、开发拆分

### 任务 2：页面入口接入

### 任务 3：排序保存

- 复用测试页组内上移/下移模型
- 保存文本记录

### 任务 4：差异提示

- 输出统计卡片
- 输出差异明细列表

### 任务 5：旧测试页处理

- 视情况保留 `/test-css` 作为回归测试页
- 或在正式接入完成后弱化其职责

## 十、验收标准

### 功能验收

1. 用户可在 Footprint 页面选择一个本地主文件夹。
2. 页面可按第一层子文件夹显示图片分组。
3. 用户可对组内图片排序。
4. 点击保存后，服务端文本记录被更新。
5. 关闭页面后再次选择同名主文件夹，排序可恢复。
6. 当目录有新增 / 缺失 / 变化文件时，页面能提示。

### 非功能验收

1. 不上传原始图片。
2. 不写数据库。
3. 不接入 `storage_files`。
4. 不依赖 AList / 云挂载历史结构。

## 十一、后续升级路径

当第一阶段稳定后，再评估：

1. 是否把主文件夹名称升级为更稳的身份键。
2. 是否引入自由布局坐标持久化。
3. 是否将文本记录迁移到正式数据库表。
4. 是否补充“未匹配目录 -> 手动映射地点”的正式交互。
