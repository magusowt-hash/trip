# 挂载网盘移除后的数据库待办

## 目标

- 当前代码已移除挂载网盘实现。
- 数据库层不直接删除历史 migration 文件，改为补“回退型 migration”和清理步骤。
- 本文档作为数据库相关唯一待办入口。

## 当前仓库现状

- 代码 schema 已回退：
  - `src/db/schema.storage.ts` 不再包含：
    - `source_type`
    - `source_ref`
    - `source_folder`
- 代码 schema 已删除：
  - `src/db/schema.cloud.ts`
- 历史 migration 仍存在：
  - `drizzle/0009_cloud_storage_extension.sql`
  - `drizzle/0010_cloud_mounts_and_logs.sql`
  - `drizzle/0011_cloud_assets.sql`

## 原则

- 不修改已经存在的历史 migration 文件内容。
- 不直接手改线上库。
- 新增一条“回退挂载网盘结构”的 migration，显式撤销这三轮结构变更。

## 待办清单

### 1. 盘点真实数据库是否执行过 0009-0011

- 检查测试库是否执行过：
  - `0009_cloud_storage_extension.sql`
  - `0010_cloud_mounts_and_logs.sql`
  - `0011_cloud_assets.sql`
- 检查线上库是否执行过上述 migration。
- 记录结果：
  - 从未执行过
  - 只在部分环境执行过
  - 已在所有环境执行过

### 2. 生成回退 migration

建议新增一条新的 migration，例如：

- `0012_remove_footprint_cloud_mount.sql`

建议内容：

- 若存在 `cloud_assets`，则删除表
- 若存在 `cloud_sync_logs`，则删除表
- 若存在 `cloud_mounts`，则删除表
- 若 `storage_files` 存在以下字段，则删除：
  - `source_type`
  - `source_ref`
  - `source_folder`
- 若存在以下索引，则删除：
  - `sf_user_source_idx`
  - `sf_user_source_ref_unique`

注意：

- SQL 需按实际数据库方言写成可安全重复执行或至少可控失败的形式。
- 若线上库已经存在依赖这些字段的数据，需要先做数据清理再删列。

### 3. 数据清理前置检查

如果任何环境已经执行过 0009-0011，先确认：

- `storage_files` 中是否已有 cloud 来源数据
- `cloud_mounts` 是否有记录
- `cloud_sync_logs` 是否有记录
- `cloud_assets` 是否有记录

若有数据，先导出审计样本并确认是否需要保留备份。

### 4. drizzle meta 清理

- 在补新 migration 后，统一更新：
  - `drizzle/meta/_journal.json`
- 如当前仓库流程要求同步 snapshot，再按实际流程更新 snapshot。

### 5. 环境执行顺序

建议顺序：

1. 本地空库验证回退 migration 可执行
2. 测试库验证
3. 确认业务代码已全部切到无挂载版本
4. 生产库执行

## 明确不做

- 不回写或篡改已有历史 migration 编号
- 不把“删除代码”理解成“数据库自动安全”
- 不在未核实环境状态前直接删表删列

## 完成标准

- 当前所有运行环境中，不再存在挂载网盘运行所需表和字段
- `drizzle` 历史链路完整
- 代码 schema 与真实数据库一致
- `status.md` 与数据库实际状态一致
