# 团队协作规则（必看）

## 目录结构

```
docs/team-work/
├── Rules.md          ← 你正在看的文件
├── maps/              ← 当前地图模块母文件夹（示例）
│   ├── status.md      ← 模块当前状态板（唯一真相入口）
│   ├── to-do list/    ← 模块级待办清单（按主题拆分）
│   ├── done list/     ← 已完成待办清单（从 to-do list 迁入）
│   ├── server/        ← 本机（服务端 / 主开发者）
│   ├── mac/           ← 成员二
│   └── pc/            ← 成员三
├── lists/             ← 其他模块母文件夹（按需新建）
│   ├── status.md
│   ├── to-do list/
│   ├── done list/
│   ├── server/
│   ├── mac/
│   └── pc/
└── ...
```

- 后续协作必须先按“模块/功能”建立**母文件夹**
- 例如本项开发的母文件夹为 `maps/`
- `server / mac / pc` 不再直接平铺在 `docs/team-work/` 下，而应放在对应母文件夹内
- 如需对模块进行长期规划、拆分问题、记录待办，统一放在模块母文件夹下的 `to-do list/`
- 每个模块必须维护一个 `status.md`，作为该模块**当前真相入口**
- `status.md` 只写当前状态，不写过程，不替代成员个人记录

## 模块状态板（status.md）

- 每个模块母文件夹下必须有 `status.md`
- `status.md` 负责记录该模块当前统一状态，至少包括：
  - 当前可用入口
  - 当前已知阻塞
  - 当前数据库 / 接口 / 文档差异
  - 当前是否允许 push
  - 当前是否完成联调
  - 当前是否完成真实环境验证
- 个人记录、to-do list、done list、database 文档如有变化，涉及模块当前真相时，必须同步更新 `status.md`

## To-Do List 机制

- `to-do list/` 用于记录某个模块的**长期待办、结构问题、重构计划、未完成事项**
- `to-do list/` 下的文件按**主题命名**，不要按时间命名
- 示例：

```text
docs/team-work/maps/to-do list/
├── 地图包模块化改造.md
├── 地图后台管理挂载方案.md
└── 铁路地图与普通地图隔离边界.md
```

- `to-do list/` 适合记录：
  - 当前实现存在的问题
  - 后续计划的模块边界
  - 需要多人协作推进的拆分事项
  - 尚未落地、但已经确认方向的方案
- 每个模块同时建立 `done list/`
- `done list/` 用于记录已经完成的待办主题，来源必须是原 `to-do list/` 中已完成的事项
- 某个 `to-do list/` 项完成后，处理方式为：
  - 将对应主题迁入 `done list/`
  - 在原 `to-do list/` 文档中对已完成事项添加“已完成”注释，标明迁移去向或完成状态
- 不允许做完后只口头说明、不更新 `to-do list/` / `done list/`
- `to-do list/` 不替代成员个人开发完成记录
- 成员个人开发完成记录仍然写在各自的 `server / mac / pc` 目录中，并继续遵守“确认后再写文档、再 push”的规则

## 协作流程

1. 各成员在自己的环境上完成分配到的开发/调试任务。
2. 各成员在模块对应的母文件夹下进行协作记录，例如：`docs/team-work/maps/mac/`
3. **一个模块的完整开发调试结束前**，各成员可先口头/聊天同步，不要急于写协作文档。
4. 每个成员完成本次开发或调试后，必须先由我（项目负责人）**确认完成**。
5. **未确认前，不要写协作文档，不要 git push。**
6. 只有在我确认完成后，该成员才需要：
   - 将本次改动摘要写入自己目录下的文档
   - 再执行 `git push` 推送远程仓库

## 完成定义

一个任务是否完成，必须按以下三段式定义判断：

### 1. 代码完成
- 本轮目标代码已写完
- 已完成最小自检
- 改动范围已明确

### 2. 运行完成
- 已完成页面 / 接口 / 脚本的实际验证
- 若未验证，必须明确写出“未验证项 + 原因”
- 不能把“代码已写完”视为“任务已完成”

### 3. 协作完成
- 已获得项目负责人确认
- 已按规则更新个人记录 / `status.md` / `to-do list` / `done list` / `database`
- 已执行 `git push`

- 只有三段都成立，才算本轮任务完成

## 文档命名规则

```
时间(精确到分)-标题.md
```

示例：
```
2026-05-16-1430-完善地图POI收藏接口.md
2026-05-16-1515-修复高德逆地理city字段空数组.md
```

- 时间格式：`YYYY-MM-DD-HHmm`（24小时制，精确到分钟）
- 标题：简要描述本次改动的主要内容
- 本规则适用于成员个人开发记录，不适用于 `to-do list/`

## To-Do List 命名规则

- `to-do list/` 下文件名直接使用主题命名
- 示例：
  - `地图包模块化改造.md`
  - `地图后台管理界面来源规范.md`
  - `铁路地图后续拆分计划.md`
- 文件名应直接体现问题域或主题，不加时间前缀

## 文档内容要求

每份文档至少包含：

```markdown
# 时间 - 标题

## 改动范围
- 列出涉及的文件

## 改动目标
本次要解决什么问题

## 改动思路
本次采用什么方案，为什么这样做

## 改动内容
实际改了什么
```

- 成员个人过程文档仅写：
  - `改动范围`
  - `改动目标`
  - `改动思路`
  - `改动内容`
- 不再强制在个人过程文档中写 `验证方式` / `注意事项`
- 验证结论、未验证项、运行状态，统一写入模块 `status.md` 或相关模块级文档

## 数据库协作规则

- 以后所有数据库相关字段、表结构、关系定义，必须以 `docs/team-work/database/` 为准
- 服务器端做任何数据库相关增删改，包括但不限于：
  - 新增字段
  - 删除字段
  - 修改字段类型
  - 调整表关系
  - 增加兼容层
  - 增加 migration
  都必须同步写入 `docs/team-work/database/`
- 任何涉及数据库的服务端改动，如果没有更新 `database/`，视为未完成
- 如果代码已经做了兼容，但数据库结构尚未统一，文档必须明确写：
  - 哪些是“代码兼容”
  - 哪些是“结构已统一”
  - 哪些仍待 migration 根治

## Git 提交

- 不要每次小改动或每轮调试后都立即 `git push`
- 必须等我确认“本次任务/本轮调试已完成”后，才允许写协作文档并执行 `git push`
- 文档写入后，再执行 `git push` 推送到远程仓库，确保其他成员可拉取到最新状态
- 提交信息格式建议：`docs(team-work): 2026-05-16-1430-完善地图POI收藏接口`
- 如本次只更新模块级 `to-do list/`，提交信息可按主题描述，例如：`docs(team-work): 更新 maps 模块化待办清单`

## Push 前检查清单

执行 `git push` 前，至少检查以下 5 项：

1. `git status` 中是否只包含本轮相关改动
2. 当前任务是否已有明确验证结论，或明确写出未验证原因
3. 是否存在数据库差异；若存在，`docs/team-work/database/` 是否已同步
4. 模块 `status.md`、必要的 `to-do list/` / `done list/` 是否已同步
5. 提交信息是否能准确反映本轮改动主题

## 注意事项

- 每个成员**只在自己所属模块的个人文件夹内写入**，不要修改其他成员的文档。
- 确认流程不可跳过：**未确认的改动不应提交文档，也不应 git push。**
- 文档尽量简洁，面向团队内部沟通，不必过度修饰。
- `to-do list/` 是模块级公共文档，允许围绕该模块方案进行维护，但内容必须聚焦结构规划、待办事项和分工，不要混写个人开发流水账。
- `done list/` 是模块级公共文档，只记录已完成待办，不写个人流水。
- 涉及数据库的服务端改动，必须同步更新 `docs/team-work/database/`，该要求高于个人习惯。

## 模块完成总结

- 每个模块开发完成后，由**项目负责人**在模块母文件夹下创建 `completed.md`
- 内容为该模块全部开发工作的最终总结，**以最新代码为准**
- 所有被修改、被废弃的方案，以 `completed.md` 中的描述为最终定稿
- `completed.md` 写入后需 `git push`，作为该模块的交付标记
