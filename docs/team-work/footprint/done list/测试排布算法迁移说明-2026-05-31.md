# 测试排布算法迁移说明

## 目标

本说明用于将本机测试页 `test/app/test-layout-v2/page.tsx` 的排布算法迁入正式页。

当前测试页的目标已收敛为两点：

1. 随机点位置固定后，末端矩形尽量不碰撞。
2. 在线条不相交的前提下，尽量避免局部过密与个别线条过长。

当前版本已经放弃了早期大量分层规则、颜色规则、gap 规则和多层级偏移规则，正式页迁移时应以当前这套“直线出射 + 末端矩形避碰 + 局部联动微调”为唯一基线，不要再混入旧测试页逻辑。

## 当前算法的核心模型

### 1. 输入

每个点仅需要：

- `id`
- `x`
- `y`

测试页当前用 `buildRandomPoints()` 生成随机点；正式页应改为真实足迹点位。

### 2. 中间数据

`LayoutEntry` 是主计算结构，核心字段：

- `pointRadius`：点到中心距离
- `angle`：点相对中心角度
- `rectWidth` / `rectHeight`：末端矩形尺寸
- `rectArea`：矩形面积
- `baseLineLength`：基础直出长度
- `lineLength`：当前实际线长
- `endX` / `endY`：线段末端位置

`EndRect` 是末端矩形结构，核心字段：

- `x` / `y` / `width` / `height`
- `anchorX` / `anchorY`

其中 `anchorX / anchorY` 是线段真正连接矩形的锚点。

## 当前排布流程

### 第一步：基础排序

入口函数：`comparePlacementOrder()`

当前规则：

1. 小面积矩形优先排。
2. 面积相同则离中心更近的点优先排。
3. 再按 `id` 稳定排序。

作用：

- 让小矩形优先占据内层。
- 大矩形更容易被推向外层。

### 第二步：基础外推排布

入口函数：`buildBaseEntries()`

当前策略：

1. 每个点沿自己的原始角度向外发射。
2. 以 `DISPLAY_RADIUS` 作为基础外推起点。
3. 对当前点做二分搜索，找到“不与已放置矩形碰撞”的最短半径。

这里的核心约束是：

- 不改点位置。
- 初始阶段不强行均分终点。
- 优先保持每条线沿原方向直出。

### 第三步：矩形碰撞处理

入口函数：`resolveRectCollisions()`

当前策略：

1. 若两个矩形碰撞，只延长其中较小面积矩形对应的线。
2. 延长方向始终沿该线自身角度。
3. 延长前检查新线段是否与其他线相交。

这一步的目标不是美观，而是先把显式碰撞消掉。

### 第四步：长度回收

入口函数：`tightenRectSpacing()`

当前策略：

1. 先找出被延长最多的线。
2. 对每条被延长的线做二分回收。
3. 在“不碰撞、不交叉”的条件下，把长度尽量收回。

这一步用于避免前一步把个别线推出极端长度。

### 第五步：单线微调

入口函数：`optimizeAngles()`

当前策略：

1. 以额外延长较多的线为优先目标。
2. 对单条线，或与相邻 1 到 2 条线组成的小联动组，尝试多个角度偏移。
3. 同时尝试多个缩短比例。
4. 每次只接受整体评分更优的结果。

### 第六步：双线联动微调

入口函数：`optimizeAngleGroups()`

当前策略：

1. 对相邻两条线作为一组。
2. 尝试同向、反向等几种偏移组合。
3. 同时尝试长度缩短比例。
4. 保留评分更优且无交叉、无碰撞的结果。

### 第七步：邻位最小间隔微调

入口函数：`refineNeighborSpacing()`

这是当前收尾阶段最重要的一层。

当前规则：

1. 计算按角度顺序排列后的相邻矩形间距。
2. 若某对邻位间距过小，则将其作为局部密集区。
3. 不是只调这两条线，而是把前后相邻线一起组成 2 到 4 条的连续段联动微调。
4. 同时尝试小角度偏移与轻微缩短。
5. 禁止生成新的最长极值线段。
6. 只接受整体评分更优的结果。

当前关键参数：

- `NEIGHBOR_MIN_GAP = 22`
- `NEIGHBOR_RELAX_PASSES = 18`
- `NEIGHBOR_ANGLE_DEGREES = [12, 8, 4, 2]`
- `NEIGHBOR_LENGTH_FACTORS = [0.88, 0.94, 0.98, 1]`

## 当前评分函数

入口函数：`computeLayoutScore()`

当前评分由以下部分组成：

1. 包围范围面积 `envelopeArea`
2. 总额外延长 `extraLength`
3. 最大额外延长 `maxExtraLength`
4. 额外延长方差 `lengthVariance`
5. 邻位最小间隔不足惩罚 `neighborGapPenalty`
6. 角度分布均衡惩罚 `angularBalancePenalty`
7. 扇区密度均衡惩罚 `sectorBalancePenalty`

其设计意图不是几何最优，而是视觉均衡：

- 不希望只为了消碰撞把少数线拉得特别长。
- 不希望局部矩形严重扎堆。
- 不希望整体外扩包围框无节制增大。

## 当前几何硬约束

以下约束是正式页迁移时必须保留的底线：

1. 点位不可修改。
2. 矩形尺寸不可修改。
3. 线段之间不可相交。
4. 矩形之间不可重叠。
5. 收敛时不能通过制造新的极端长线来换取局部空隙。

对应的几何函数：

- `rectsOverlap()`
- `segmentsIntersect()`
- `lineIntersectsAtIndex()`
- `hasAnyLineIntersections()`

## 末端矩形锚点规则

入口函数：`buildRectForEntry()`

当前规则：

- 下方区域：以矩形顶部中心为线段连接点
- 上方、左侧、右侧区域：以矩形底部中心为线段连接点

当前代码中通过 `entry.endY > 0` 判断是否属于下方区域。

如果正式页后续需要更精细的四象限规则，可以在这里扩展，但不要改动后续碰撞与优化函数的输入输出格式。

## 测试页中仅用于观察的内容

以下内容不是正式页必须迁移的核心算法：

- 黄色空隙格：`buildGapCells()`
- 外层包围框：`buildRectEnvelope()`
- 右侧测试面板
- 随机点生成 UI

这些都属于调试辅助视图。正式页可去掉，或仅保留开发开关。

## 建议的正式页迁移方式

### 方案

不要直接把整个测试页复制进正式页。应拆成纯算法层和渲染层。

建议拆分为：

1. `layout-types.ts`
2. `layout-geometry.ts`
3. `layout-score.ts`
4. `layout-engine.ts`
5. 正式页组件

### 纯算法层建议拆分

`layout-types.ts`

- `Point`
- `LayoutEntry`
- `EndRect`
- `BoundsRect`

`layout-geometry.ts`

- `distance`
- `normalizeAngle`
- `projectAngle`
- `rectsOverlap`
- `segmentsIntersect`
- `computeRectEdgeDistance`

`layout-score.ts`

- `computeAngularBalancePenalty`
- `computeSectorBalancePenalty`
- `computeLayoutScore`

`layout-engine.ts`

- `comparePlacementOrder`
- `buildBaseEntries`
- `resolveRectCollisions`
- `tightenRectSpacing`
- `optimizeAngles`
- `optimizeAngleGroups`
- `refineNeighborSpacing`
- `buildLayout`

正式页组件仅负责：

- 将真实点位转成 `Point[]`
- 提供真实矩形尺寸
- 调用 `buildLayout()`
- 按返回值渲染线与矩形

## 正式页接入时必须替换的测试逻辑

### 1. 随机点

替换 `buildRandomPoints()`，改为真实数据源。

### 2. 伪随机矩形尺寸

替换 `buildRectSize()`。

测试页现在是基于 `seed + id` 的伪随机尺寸生成：

- `RECT_SIZE_SEED_X`
- `RECT_SIZE_SEED_Y`
- `RECT_SIZE_ID_X`
- `RECT_SIZE_ID_Y`

正式页应改为真实图片卡片尺寸，来源建议为：

1. 图片原始宽高比
2. 业务规定的卡片宽高范围
3. 缩略图/原图展示模式

### 3. 调试视图

以下建议默认不迁入正式页：

- 空隙格
- 包围框
- 测试指标面板

如需保留，仅通过开发开关显示。

## 建议的正式页接入顺序

1. 先抽出纯算法函数，不改行为。
2. 用正式页真实点位替换随机点。
3. 用正式页真实卡片尺寸替换伪随机矩形尺寸。
4. 仅渲染线与矩形，先不接入复杂交互。
5. 用正式页实际数据观察极端场景。
6. 最后再决定是否增加业务侧参数开关。

## 已知不足

当前算法已经比旧测试规则稳定得多，但仍有以下边界问题：

1. 它是启发式收敛，不是严格全局最优。
2. 当单一区域矩形数量极端集中时，可能仍会出现局部线长偏长。
3. `comparePlacementOrder()` 当前仅按面积和点距排序，尚未引入更强的方位密度预判。
4. `entry.endY > 0` 的上下区域锚点判断较粗。
5. 当前评分依赖经验权重，正式页真实数据接入后可能需要重新调参。

## 迁移时禁止回退的事项

正式页迁移时，不要重新引入以下旧方向：

1. 多层颜色规则驱动的复杂偏移链。
2. 手工维护的多重 gap 语义。
3. 超长规则链式优先级。
4. 先做大量局部特判，再补几何修复。

原因很明确：这些旧逻辑已经在测试过程中证明过于脆弱，难以维护，也不利于正式页稳定接入。

## 当前建议

正式页迁入时，应把这套算法视为：

- 一个独立的“末端卡片排布引擎”
- 输入为点位和卡片尺寸
- 输出为线段终点与卡片位置

不要把它和地图业务、筛选逻辑、弹窗状态、图片加载状态混在一个组件里实现。应保持单独的纯计算边界，便于后续继续调参与做极端场景回归。
