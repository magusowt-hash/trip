# 本地目录索引方案 - 歧义点澄清

## 一、目的

当前 `footprint` 已确认不继续沿用“挂载网盘 / AList”方案，转为“浏览器选择本地主文件夹 -> 前端读取 -> 服务端仅记录索引与排序”的方向。  
在进入正式开发前，必须先把容易反复摇摆的歧义点定清楚。

以下内容即为第一阶段默认口径。

## 二、歧义点与结论

### 1. “路径”到底指什么

歧义：

- 用户口头上说“按路径识别”
- 浏览器文件选择器无法稳定提供真实绝对路径

结论：

- 第一阶段不以真实绝对路径作为真相源
- 第一阶段以“主文件夹名称”作为记录键
- 用户再次选择目录后，如果主文件夹名称相同，则命中同一份记录

说明：

- 这不是最终最稳的技术身份
- 这是当前浏览器能力与测试结论下的最低可行规则

### 2. 不同真实路径但主文件夹同名怎么办

歧义：

- 例如两个不同磁盘里都存在 `JapanTrip`

结论：

- 第一阶段接受“同名即同记录”
- 即：不同真实路径但主文件夹同名，会复用同一份排序与设置

说明：

- 这是已知风险，不是 bug
- 若后续业务不能接受，再升级成“主文件夹名称 + 人工标识”或“目录句柄指纹”模式

### 3. 什么叫“位置”

歧义：

- 是画布坐标
- 还是列表顺序
- 还是分组内排序

结论：

- 第一阶段“位置”只表示排序位置
- 不落 `frame_x / frame_y`
- 不做自由布局坐标持久化
- 先只支持组内上移 / 下移形成稳定排序号

说明：

- 这是为了降低正式接入风险
- 真正的自由布局若要做，应另起一轮方案，不要混在第一阶段

### 4. 子文件夹与地点名的匹配规则是什么

歧义：

- 完全相等
- 模糊匹配
- 去空格匹配
- 忽略大小写

结论：

- 第一阶段采用“原始名称完全一致优先”
- 可补一个轻量标准化规则：
  - 去掉首尾空格
  - 连续空格压成一个空格
- 不做拼音、别名、模糊命中

说明：

- 第一阶段必须让规则简单可解释
- 命不中就提示，不自动猜

### 5. 子文件夹层级支持到哪一层

歧义：

- 只看第一层
- 递归扫描所有子目录

结论：

- 第一阶段只认主文件夹下第一层子文件夹
- 第一层子文件夹即地点目录
- 更深层级仅作为该地点目录内部文件结构的一部分，不再继续推导地点

说明：

- 例如 `JapanTrip/东京/day1/a.jpg`
- 地点只认 `东京`
- `day1` 不再作为二级地点

### 6. 根目录下直接放图怎么办

歧义：

- 主文件夹根下可能直接有图片，不在任何子文件夹内

结论：

- 第一阶段允许读取，但归入 `根目录未分类`
- 不自动映射到任何足迹项
- 正式 Footprint 页面中默认只提示，不进入已匹配地点展示

### 7. 非图片文件怎么处理

结论：

- 第一阶段只读取图片扩展名白名单
- 非图片文件忽略
- 后续若需要，可在提示区显示“已忽略 N 个非图片文件”

### 8. 文件身份如何判定“同一张图”

歧义：

- 看文件名
- 看相对路径
- 看内容 hash

结论：

- 第一阶段用 `relativePath + size + lastModified` 作为差异判断主规则
- 排序位置的复用键先用 `relativePath`

说明：

- 这意味着目录改名、文件移动时，第一阶段会视为新增/缺失
- 这是可接受的简化

### 9. 差异提示展示到什么粒度

结论：

- 第一阶段只要求三类：
  - 新增文件
  - 缺失文件
  - 已变化文件
- 不要求进一步识别“重命名”“移动目录但内容相同”

### 10. 排序真相源在哪

结论：

- 第一阶段排序真相源在服务端文本记录
- 前端只维护本次会话临时状态
- 刷新后重新选择同名主文件夹，再从文本记录恢复

### 11. 是否接数据库

结论：

- 第一阶段不接数据库
- 只使用服务端文本记录
- 数据库设计单独后置

说明：

- 当前重点是先把 Footprint 正式页面中的业务链路跑通
- 不是先做重型持久化基础设施

### 12. 是否进入现有 `storage_files`

结论：

- 不进入
- `storage_files` 继续只承接真正上传到服务端的图片
- 本地目录索引图片不复用 `storage_files`

### 13. 用户再次进入为什么还要重新选择目录

结论：

- 因为浏览器不会把本地目录访问权自动变成长期后台访问能力
- 第一阶段产品文案必须明确：
  - “重新进入后，需要再次选择主文件夹以校验目录内容并恢复排序”

### 14. 多个主文件夹是否允许并存

结论：

- 第一阶段文本记录层允许并存多个主文件夹名称记录
- 但单次页面只操作一个当前主文件夹

### 15. 页面设置包括哪些

结论：

- 第一阶段页面设置只包含与当前图片索引视图直接相关的轻量项
- 建议包含：
  - 当前排序结果
  - 是否展开某些分组
  - 基础展示开关
- 不把全量 `footprint` 外框设置都并入这套文本记录

## 三、本轮定稿原则

第一阶段正式开发时，一律按以下原则执行：

1. 简单规则优先，不做隐式猜测。
2. 主文件夹名称就是当前记录键。
3. 相对路径是当前排序复用键。
4. 排序位置仅表示顺序，不表示自由坐标。
5. 只做文本记录，不动数据库。

## 四、后续可能升级点

以下内容明确不属于第一阶段，但可以作为第二阶段候选：

- 主文件夹名称 + 人工别名双键
- 目录句柄指纹
- 自由布局坐标持久化
- 更复杂的地点名匹配规则
- 目录移动/重命名识别
- 正式数据库化
