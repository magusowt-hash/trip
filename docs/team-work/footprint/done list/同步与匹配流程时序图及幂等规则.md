# 同步与匹配流程时序图及幂等规则

## 目标

- 定义第一阶段云端资源接入的完整执行流程
- 明确同步、匹配、展示、提示之间的时序关系
- 提前收口重复同步、目录改名、文件删除、重新匹配等高风险边界

## 适用前提

- 用户手动触发同步
- 只扫描网盘根目录第一层文件夹
- 一级文件夹名直接匹配当前足迹地点列表
- 文件夹匹配成功后，该文件夹下所有图片自动归入该地点
- 未匹配资源入库但不进入正式 Footprint，只做提示

## 参与对象

- 用户端 Footprint 页面
- 管理端提示页面
- `CloudRootSyncService`
- `FootprintFolderMatchService`
- 云端 provider 访问层
- 云端资源数据表

## 主流程时序

```text
用户
  -> Footprint 页面点击“同步网盘”
  -> POST /api/footprints/cloud/sync

API
  -> CloudRootSyncService.syncCloudRoot(userId)
  -> 读取用户 root 配置
  -> 调 provider 列出根目录第一层文件夹
  -> 对每个文件夹列出图片
  -> upsert cloud_asset_folders
  -> upsert cloud_assets
  -> 标记本轮 last_seen_at
  -> 记录 sync log

API
  -> FootprintFolderMatchService.matchFoldersToFootprints(userId, rootId)
  -> 读取当前用户足迹地点列表
  -> 用 folder_name 精确匹配 place_title
  -> matched: 写 folder.match_status=matched
  -> matched: 为该目录下全部图片 upsert binding
  -> unbound: 写 folder.match_status=unbound
  -> unbound: 不写 binding

API
  -> 返回同步摘要

页面
  -> 刷新 GET /api/footprints/cloud/outer-frame
  -> 刷新 GET /api/footprints/cloud/hints
```

## 读流程时序

### 已绑定云端资源

```text
页面
  -> GET /api/footprints/cloud/outer-frame

API
  -> 查 footprint_cloud_bindings
  -> join cloud_assets
  -> left join footprint_view_projections
  -> 过滤可访问资源
  -> 按 place_title 聚合
  -> 返回给页面
```

### 未匹配提示

```text
页面 / 管理端
  -> GET /api/footprints/cloud/hints

API
  -> 查 cloud_asset_folders
  -> 过滤 match_status=unbound
  -> 取 asset_count + sample_asset_id
  -> 返回提示数据
```

## 幂等设计原则

### 原则 1

同一个用户、同一个 root、同一个相对路径的目录或文件，重复同步不能产生重复记录。

### 原则 2

同步是“对远端当前状态做收敛”，不是“无条件追加新记录”。

### 原则 3

匹配结果应可重算，不依赖一次性写死逻辑。

### 原则 4

未匹配资源必须保留，以便后续地点列表变化后重新匹配。

## 幂等键建议

### 目录

以：

- `root_id + relative_path`

作为唯一键。

说明：

- 第一阶段只扫一级目录，因此该键稳定且足够

### 资源

以：

- `root_id + relative_path`

作为主幂等键。

辅助识别字段：

- `provider_file_id`
- `etag`
- `checksum`

说明：

- 第一阶段不要求 provider 一定提供稳定 file id
- 主流程先用路径收敛已足够

### binding

以：

- `asset_id`

作为唯一绑定键。

说明：

- 第一阶段一个资源只允许归属一个地点

### projection

以：

- `asset_id + view_type + layout_key`

作为唯一键。

## 同步执行规则

### 第一步：建立本轮同步标记

在开始扫描前生成：

- `sync_run_id`
- `started_at`

说明：

- 所有本轮看见的目录和文件都更新 `last_seen_at = started_at`
- 后续可用来判断“本轮未再出现”的记录

### 第二步：扫描一级目录

对于 provider 返回的每个一级目录：

- upsert `cloud_asset_folders`
- 更新 `folder_name`
- 更新 `relative_path`
- 更新 `last_scanned_at`
- 更新 `last_seen_at` 语义到目录级

### 第三步：扫描目录下图片

对于每个图片文件：

- 若不存在：insert `cloud_assets`
- 若已存在：update 元数据
- 统一更新：
  - `file_name`
  - `size`
  - `etag`
  - `thumb_ref`
  - `origin_ref`
  - `last_seen_at`
  - `deleted_at = null`
  - `access_state`

### 第四步：同步完成后处理缺失记录

对属于该 `root_id` 且本轮未被看到的资源：

- 不直接物理删除
- 标记：
  - `access_state = missing`
  - `deleted_at = now`

对本轮未被看到的目录：

- 若目录下所有资源都 missing，可保留目录记录并更新提示状态
- 不建议立即删除目录记录

说明：

- 第一阶段优先保留历史痕迹，避免误删误判

## 匹配执行规则

### 规则 1：匹配来源

只使用：

- `cloud_asset_folders.folder_name`
- 当前用户足迹地点列表中的地点名

不使用：

- 子目录名
- 文件名
- 全站地点库

### 规则 2：匹配方式

第一阶段采用精确匹配。

建议标准化后再比对：

- trim 首尾空格
- 连续空格压缩
- 全角半角统一

说明：

- 标准化只用于比较
- 原始名称仍保留存储

### 规则 3：匹配成功

若目录名命中地点：

- `cloud_asset_folders.match_status = matched`
- `matched_place_title = 目标地点名`
- 对该目录下全部 `cloud_assets` upsert `footprint_cloud_bindings`

### 规则 4：匹配失败

若未命中：

- `cloud_asset_folders.match_status = unbound`
- `matched_place_title = null`
- 删除或失效该目录下旧 binding

说明：

- 这样可保证重新同步后，目录状态总是与当前地点列表一致

## 重新同步场景

### 场景 1：重复点击同步

预期：

- 不产生重复目录
- 不产生重复资源
- 不产生重复 binding
- 只更新最新元数据和统计

处理：

- 依赖 upsert + 唯一键完成

### 场景 2：新增目录

预期：

- 新目录和新图片入库
- 若目录名命中地点，直接进入已绑定展示
- 未命中则进入提示

### 场景 3：目录内容新增图片

预期：

- 新图片追加到同一目录
- 若目录已 matched，新图片自动获得 binding

### 场景 4：目录被删除

预期：

- 目录下资源标记为 `missing`
- 这些资源不再进入正式展示
- 历史 projection 可保留，但不参与当前响应

### 场景 5：图片被删除

预期：

- 单张资源标记为 `missing`
- binding 保留也可以，但视图查询时必须过滤不可访问资源

推荐：

- 第一阶段保留 binding，靠 `access_state != ok` 过滤展示

## 目录改名场景

### 当前阶段建议

将目录改名视为：

- 旧目录消失
- 新目录出现

也就是：

- 旧 `relative_path` 对应目录未再出现，后续资源标记 `missing`
- 新 `relative_path` 生成新的目录记录和资源记录

原因：

- 第一阶段主幂等键就是路径
- 不引入复杂“目录移动识别”逻辑更稳

### 影响

- 若用户把 `故宫` 改名成 `北京故宫`
- 系统会认为旧目录失效，新目录待重新匹配

这符合第一阶段简化模型。

## 地点列表变化场景

### 场景 1：原来未匹配，后来地点新增

预期：

- 用户再次手动同步后
- 原 `unbound` 目录重新参与匹配
- 若命中，则转为 `matched`

### 场景 2：原来已匹配，后来地点被删除或改名

预期：

- 下一次同步重新匹配时失效
- 目录转回 `unbound`
- 原 binding 应失效或删除

推荐：

- 第一阶段直接删除该目录下 binding，再由目录状态表达未匹配

## binding 幂等规则

### 创建

当目录 `matched` 时：

- 遍历目录下全部 `asset`
- 对每个 `asset_id` upsert 一条 binding

### 更新

若同一资源仍属于同一目录、同一地点：

- 只更新时间，不新建

### 失效

当目录从 `matched` 变为 `unbound` 时：

- 删除该目录下全部 binding

原因：

- 第一阶段 binding 完全由目录匹配派生
- 不需要独立维护复杂生命周期

## projection 幂等规则

### 创建

资源第一次进入 `outer_frame` 时，projection 可为空，不强制立即创建。

### 更新

当用户拖动云端图片时：

- upsert `asset_id + view_type + layout_key`

### 保留

即使资源短期 `missing`，projection 也可保留。

原因：

- 文件恢复时可以继承历史布局

### 过滤

视图查询时：

- 如果 `access_state != ok`
- 不返回该资源给正式展示

## 未匹配提示规则

### 提示对象

只按目录提示，不按单张图片逐条提示。

返回建议包含：

- `folder_name`
- `asset_count`
- `sample_thumbnail_url`
- `reason=no_place_match`

### 提示时机

- 用户同步完成后立即提示
- 用户再次进入页面时仍可看到未处理提示
- 管理端可查看指定用户未匹配目录

## 错误恢复规则

### provider 短暂失败

处理：

- 本轮同步记 `failed`
- 不清空旧数据
- 不将未扫描到的数据直接标记 missing

原因：

- 否则一次网络失败会误伤全部资源

### 部分目录扫描失败

处理：

- 已成功扫描部分正常写入
- 失败目录记录到 sync log 的摘要信息
- 不对失败目录旧资源做 missing 收敛

### 匹配阶段失败

处理：

- 同步结果保留
- match 失败单独报错
- 下次可只重跑匹配逻辑

说明：

- 同步和匹配逻辑要尽量可拆分重试

## 推荐事务边界

### 不建议

- 整个同步过程包成一个超大事务

原因：

- 目录和图片较多时风险高
- provider 调用本身也不适合长事务

### 建议

- 以单目录为小事务单位
- 每个目录扫描完成后提交：
  - folder upsert
  - assets upsert
  - folder 级 match / binding 更新

好处：

- 失败影响面小
- 可局部重试

## 第一阶段推荐实现口径

### 同步

- 按 root 扫描
- 按目录提交
- 成功目录立即可用

### 匹配

- 目录扫描后立即匹配
- 目录状态是唯一真相源

### 展示

- 只展示 `binding 存在 && asset.access_state = ok`

### 提示

- 只提示 `folder.match_status = unbound`
